# 附加参考案卷 (MDM 及防雪崩底座法则)

为了帮助你在判案时有充分的“判例”和“法理”依据，以下摘录了自 `contexts.md` 等架构文件中的部分防御性法则，当你不确定时可直接引用作为对用户的反驳：

1. **Surrogate PK (无业务含义主键) 的护城河**

   - **法条原文**：系统坚决不允许将外部数据源的自带 ID 作为我们库的 `id`（物理主键）。必须全部将其转换为补充字符串字段（降级为类似 `gitlab_id: str`, `zentao_id: str`）。我们的物理 ID 永远只能是由我们自己的 ORM 独立签发的无含义自增或 UUID。
   - **为什么要这么做？** 这个底座用来防止各种不同厂商发过来的 ID 类型和主外键冲突，也是 OneID 生效与不被绑架的前提。

1. **循环外键的延后挂载法宝 (Two-Phase Protocol 的组成)**

   - **法条原文**：如果是双向互锁的业务强模型约束（如 User 挂载 Location 且 Location 管理员为某 User 这种奇葩需求）。
   - **技术补丁**：在写 SQLAlchemy 代码时，其中一个实体的 ForeignKey(xx.id) 内定义参数必须携带 `use_alter=True`。这是为了保护 SQLite 测试和建表器能在第一步创建两张空表，在第二步反向去给这俩表挂载锁。避免跑测试死锁！

1. **审计引擎是副业不能牵连主业**

   - **法条原文**：审计监听或写队列的任何行为：(1) 外围必须裹着独立 `try...except` 消化崩溃。(2) 解析 Diff 计算新旧值交替时，严厉禁绝提取 `Relationship` 数据树进行包裹，否则它会拉走底下所有的附属层级，这相当于一次业务调用拉来了整座冰山的级联查询。
