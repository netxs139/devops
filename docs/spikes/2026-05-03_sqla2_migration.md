## 1. 背景与目标 (Context & Objectives)

为了消除 `base_models.py` 中的类型推断错误，并利用 SQLAlchemy 2.0 的强类型特性，我们需要将现有的 `Column` 定义迁移为 `Mapped` 声明。本次调研旨在验证：

1. `TimestampMixin` 和 `SCDMixin` 与 `Mapped` 风格的兼容性。
1. 跨模块外键循环（如 `Organization.manager_user_id`）在 2.0 下的处理方式。
1. 自引用关系（Parent/Child）的迁移路径。

## 2. 实验方案 (Experimental Design)

- **验证环境**: `scratch/spike_sqla2_migration.py` (使用 SQLite 内存库)
- **核心假设**: 假设使用 `mapped_column` 可以完美替代 `Column` 并保持 `default/nullable` 等元数据行为。

## 3. 证据与发现 (Evidence & Findings)

通过编写实验脚本并多次修正，得出以下结论：

### 3.1 Mixin 兼容性

```python
class TimestampMixin:
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), default=lambda: datetime.now(UTC)
    )
```

验证通过。`mapped_column` 可以承载所有的旧版 `Column` 参数。

### 3.2 关系歧义风险 (AmbiguousForeignKeysError)

在 2.0 强类型模式下，由于 `TimestampMixin` 引入了 `created_by` 外键，而业务模型（如 `Organization`）又引入了 `manager_user_id` 外键，SQLAlchemy 无法自动推断关系路径。

- **解决方案**: 必须在 `relationship` 中显式指定 `foreign_keys` 或 `primaryjoin`。

### 3.3 自引用迁移

```python
parent: Mapped[Optional["Organization"]] = relationship(
    "Organization", remote_side="Organization.id", back_populates="children"
)
```

验证通过。

## 4. 实施建议 (Implementation Advice)

1. **类型别名**: 建议在 `base_models.py` 顶部定义 `int_pk = Annotated[int, mapped_column(primary_key=True)]` 以减少重复。
1. **强制显式关系**: 所有存在多个外键指向同一表的 Relationship，必须补全 `foreign_keys` 参数。
1. **分步替换**:
   - 第一步：迁移 Mixins。
   - 第二步：迁移核心实体（Organization, User）。
   - 第三步：迁移关联表。
