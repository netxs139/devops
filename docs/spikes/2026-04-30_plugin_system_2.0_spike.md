## 1. 现状痛点分析 (Current Issues)

目前插件系统采用“物理导入即注册”模式：

- **副作用风险**：`from .worker import GitLabWorker` 在 `__init__.py` 中会触发大量依赖导入，即使我们只是想查询插件名称。
- **循环依赖**：如果 `GitLabWorker` 需要反向引用 `core` 中的某些动态组件，而 `core` 又在加载插件，就会导致导入崩溃。
- **缺乏治理能力**：无法在不导入整个业务逻辑的情况下进行“健康检查”或“配置预校验”。
- **模型加载混乱**：`Base.metadata` 的收集完全依赖于导入顺序。

## 2. 目标架构 (Target Architecture: Plugin 2.0)

### 2.1 核心组件定义

1. **`BasePlugin` (Interface)**: 定义插件的身份、依赖和能力。
1. **`PluginMetadata` (Value Object)**: 包含名称、版本、支持的数据源类型、所需配置 Schema 等。
1. **`PluginManager` (Orchestrator)**: 负责双阶段加载：
   - **Phase 1: Discovery**: 仅读取 `__init__.py` 中的 `plugin` 对象元数据。
   - **Phase 2: Execution**: 根据需要实例化 `Worker` 或 `Client`。

### 2.2 生命周期钩子 (Lifecycle Hooks)

- `on_load()`: 插件被发现时的轻量级动作。
- `on_setup()`: 资源初始化（如建立连接池）。
- `health_check()`: 物理环境连通性验证。
- `on_teardown()`: 资源释放。

## 3. 技术原型验证 (PoC Draft)

### 3.1 插件定义示例 (`gitlab/__init__.py`)

```python
from devops_collector.core.base_plugin import BasePlugin, PluginMetadata

class GitLabPlugin(BasePlugin):
    metadata = PluginMetadata(
        name="gitlab",
        version="2.0.0",
        data_source_type="git_scm",
        required_config=["url", "token"]
    )

    def get_worker_class(self):
        # 延迟导入，避免 Side-effect
        from .worker import GitLabWorker
        return GitLabWorker

    async def health_check(self, config) -> bool:
        # 在不启动采集的情况下验证 API 通讯
        from .gitlab_client import GitLabClient
        return await GitLabClient(config).ping()

# 唯一暴露的入口
plugin = GitLabPlugin()
```

## 4. 迁移策略 (Migration Strategy)

1. **兼容层**：重写 `PluginLoader` 使其同时支持旧的 `Side-effect` 模式和新的 `BasePlugin` 模式。
1. **核心层重构**：创建 `devops_collector/core/base_plugin.py`。
1. **分批迁移**：优先迁移复杂的 `gitlab` 和 `sonarqube` 插件。
