# 第一章 项目背景与使用场景

## 1.1 组织现状

公司拥有7个研发中心（约300人）和10个交付中心（约700人），总人数约1000+。各研发中心通过公司内网Nexus管理自研制品(Maven/PyPI/NPM/Docker)，同时共享一个公共代理库拉取外部第三方依赖。交付中心人员需要从研发中心获取产品包，部署到客户现场。当前制品流通依赖人肉通道——研发人员打包发邮件或共享网盘，效率低且不可追溯。

## 1.2 核心痛点

- **研发→交付制品传递断层**：Nexus/JFrog 解决的是"研发团队内部的制品仓库管理"，我们的痛点是"研发→交付的制品传递断层"
- **非技术人员无法使用现有工具**：现有产品的 UI/UX 面向开发者（Maven坐标、GAV、SNAPSHOT术语），交付人员只需要"搜→看→下"三件事
- **组织架构适配困难**：7研发中心+10交付中心的矩阵结构、产品跨中心划转——是公司独有的组织逻辑，商业产品无法原生支持
- 交付人员无法直观浏览"公司有哪些产品、最新版本是什么"
- 制品分散在多个Nexus仓库，下载依赖研发人员口口相传
- 缺乏按部门的权限管控——谁该下载什么产品无人管理
- 下载行为无记录，无法统计各产品/部门的实际使用情况
- 产品跨研发中心联合发布时，组件版本对应关系混乱
- 组织架构变更(产品划转)时，制品权限调整困难，依赖链路可能断裂
- 紧急下线(如安全漏洞版本)缺乏统一管控和通知能力

## 1.3 用户画像

### 超级管理员（1-2 人）

- **是谁**：IT 部门负责人或质量部门负责人
- **日常**：管理全局部门架构、处理跨中心事务、查看全局统计
- **痛点**："各中心各自为政，我不知道谁在下载什么"
- **使用频率**：每周 2-3 次，主要在月初看统计
- **关键操作**：创建/禁用部门、任命中心管理员、审计日志抽查

### 研发中心管理员（每中心 1 人，共 7 人）

- **是谁**：各研发中心负责制品发布的组长
- **日常**：审核新产品、创建发布包、管理授权
- **痛点**："交付中心天天问我最新版本在哪，我得挨个找"
- **使用频率**：每周 5-10 次，每次有发版时频繁操作
- **关键操作**：产品审定发布、LLM 辅助生成发版说明、授权管理

### 交付人员（10 个交付中心，约 700 人）

- **是谁**：到客户现场部署系统的现场工程师
- **日常**：出差前下载最新版产品包，到现场 CLI 部署
- **痛点**："发邮件要版本、等回复、U 盘拷、文件损坏重来——流程太慢了"
- **使用频率**：每周 1-5 次，集中在出差前
- **关键操作**：产品搜索浏览、Web/CLI 下载、查看通知
- **技术水平**：会 CLI 操作，不需要 IDE，不需要懂 Maven

### 交付部门管理员（10 个交付中心，每中心 1 人）

- **是谁**：交付中心项目经理
- **日常**：接收发布通知、管理本部门人员
- **痛点**："新产品发布了我不知道，出漏洞了不知道"
- **使用频率**：被动接收通知，主动操作少
- **关键操作**：查看通知、管理本部门用户

## 1.4 业务场景与流程

### 场景一：研发中心发布产品

研发人员通过IDE(settings.xml/pip.conf/Docker CLI)将制品push到本中心的Nexus仓库。制品管理定时同步自动发现新版本，状态标记为"待审定"。研发中心管理员登录管理后台审核确认后点击"发布"，产品即时出现在下载门户。研发端操作零变化——仍然push Nexus，不做任何拦截，完全透明接入。

### 场景二：交付人员下载部署

交付人员登录下载门户，搜索产品名，浏览版本列表(含更新日志)，点击下载。系统自动从多个Nexus仓库拉取对应组件的指定版本，打包为一个ZIP文件返回。紧急场景下可通过CLI脚本一键下载部署(curl|bash)，无需浏览器。

### 场景三：管理员授权与监控

超级管理员或研发中心管理员登录管理后台，将产品按需授权给交付部门。后台提供下载统计看板：哪个产品下载量最大、哪个交付部门最活跃、版本分布趋势。所有管理操作(授权、发布、下架、划转)均有完整审计日志。

### 场景四：产品跨部门划转

产品A原归属部门A，因组织调整划归部门B。管理员在后台修改产品归属为部门B，新版本发到部门B仓库。Maven groupId保持不变(所有依赖方无需修改pom.xml)，自动更新Nexus Content Selector权限规则。交付部门对划转完全无感。历史下载统计数据跟随产品归属新部门。

### 场景五：版本紧急下线

研发中心发现产品某版本存在安全漏洞。研发中心管理员在管理后台将该版本标记为"已下线"。系统立即生效——下载门户隐藏该版本、已下载过该版本的交付部门首页弹出红色紧急通知要求升级。整个过程1分钟内完成，无需通知每个交付部门或手动干预Nexus。

## 1.9 amdp定位：管理为基，分发为用

在任何规模的组织中，制品生命周期天然分为两个阶段——**入仓与出仓**。研发将制品推入仓库，交付从仓库取出制品。Nexus 社区版已成熟解决了\*\*"入仓"**（存储、代理、版本控制），但**"出仓"\*\*（谁能取、怎么取、取了什么、什么时候该升级）仍是一片空白。amdp 正是在 Nexus 之上，补全这缺失的一环。

### 制品管理——存得下、管得住、找得到

制品管理指对二进制文件（JAR、Docker 镜像、PyPI 包等）的全生命周期内部控制，核心是"存得下、管得住、找得到"。

1. **集中存储与版本控制**：通过 Hosted 仓库统一存放团队自研制品，严格区分 Release（正式版）与 Snapshot（快照版），防止版本覆盖或混乱。
1. **依赖缓存与代理**：通过 Proxy 仓库缓存 Maven Central、Docker Hub 等公共源资源，加速构建并隔离外部网络风险。
1. **安全与权限管控**：实施基于角色的细粒度访问控制，限制上传与下载权限，确保入库制品可控。
1. **元数据与清理**：记录制品的上传者、时间、校验和等元数据，并配置自动清理策略，优化存储成本。

Nexus 社区版在这四个方面已基本完善。amdp 不重复建设存储层，而是**直接复用 Nexus 的数据和制品**，通过 REST API 读取仓库信息。

### 制品分发——取得快、给得准

制品分发指将管理好的制品安全、高效地交付给下游消费者，核心是"取得快、给得准"。

1. **统一访问入口**：通过 Group 仓库聚合多个 Hosted 和 Proxy 仓库，为客户端提供单一 URL 地址。但对交付部门而言，Maven 的 `settings.xml` 毫无意义——他们需要的是一个浏览器就能打开的下载门户，搜索产品名、点按钮、拿 ZIP。
1. **高效传输与流式封装**：下游请求时优先返回本地缓存，避免重复从公网下载。amdp 在此之上构建了一条**无状态流式管线** (边下边压边传)——按产品的一次发布，从多个 Nexus 仓库建立异步 HTTP 连接，借助 aiozipstream 实时将接收到的组件数据块 (Chunk) 压缩并推入 HTTP 响应流。无论产品包体积多大，单次请求的内存消耗永远被固定在几 MB 级别，让交付端不仅能"一个文件搞定"，还彻底消除了中心服务器的内存暴涨风险。
1. **环境晋级与推送**：通过 CI/CD 流水线将制品从开发环境自动晋级至测试或生产仓库。amdp 不介入晋级流程本身，但通过**发版状态标记**（待审定→已发布→已下线）和**自动通知**，让晋级结果即时触达所有相关方。
1. **多格式兼容交付**：同时支持 Maven、Docker、PyPI、NPM 等多种协议。amdp 在此之上提供**Web 下载 + CLI 脚本**双通道，无论交付部门在办公室还是客户现场，都能一键获取。

简言之，制品管理侧重**仓库内部的秩序与安全**，制品分发侧重**对外服务的效率与统一**。Nexus 做到了前者，amdp 补上了后者。

### amdp 的补位清单

在两套能力框架之下，amdp 在 Nexus 社区版之上新增了八项关键能力：

1. **产品目录与分类**：从 Nexus 仓库中自动发现产品，按归属中心、业务类型分组，提供可视化的产品目录浏览——解决"公司到底有哪些产品"的信息盲区。
1. **部门级授权矩阵**：按交付部门授权下载特定产品，非 Nexus 仓库级 RBAC，而是产品级、部门级的细粒度授权——解决"谁该下载什么无人管"的权限真空。
1. **发版说明自动生成**：集成 DeepSeek 大模型，读取 GitLab commit 记录自动预填 release_notes，审核确认后发布——解决"发版说明靠手写、版本历史靠口述"的元数据缺失。
1. **站内通知 + 邮件推送**：新产品发布、版本下线、安全漏洞均自动推送通知给已授权部门，含 PDF 发版说明附件——解决"发布了没人知道、出漏洞无人响应"的信息断层。
1. **下载统计看板**：按产品、部门、时间三个维度统计下载量，角色隔离（中心管理员只看自己中心），为效能度量提供数据基石——解决"谁在用、用多少、什么版本最热门"的度量黑洞。
1. **CVE 漏洞追踪**：对接 GitLab CI Trivy 扫描结果，自动关联发布包→追溯下载用户→红色紧急通知+邮件——填补 Nexus 社区版无漏洞扫描的关键缺失。
1. **全操作审计日志**：所有管理操作（发布、授权、下架、划转）记录前后 JSON 快照，合规可追溯、误操作可回滚——解决"谁在什么时候做了什么"的问责真空。
1. **CLI 一键部署**：交付人员在客户现场无需浏览器，`curl | bash` 一行命令完成下载部署——解决"现场环境只有终端"的最后一公里。

> 这八项能力并非技术炫技，每一项都对应一个真实存在的组织痛点。amdp 不做 Nexus 能做的事，只做 Nexus 做不到的事。

### 定位收束

Nexus 是制品的**单一可信源**——所有制品存储于此，版本唯一、来源可溯。amdp 是单一可信源的**对外窗口**——让需要制品的人，不必懂 Maven、不必问研发、不必翻邮件，就能找到、拿到、用上。

两者结合，形成从"研发 push 制品"到"交付拿到产品包"的**完整闭环**。不替代 Nexus，不重复造轮子——在管理之上，做分发的事。

### 2.5.2 详细功能说明

#### 2.5.2.1 元数据增强类(2 项)

**AI-01 发版说明自动生成(P0)**
管理员创建发布时,系统自动从 GitLab 拉取 commit 记录,调用 DeepSeek 模型生成三段式发版说明(新增/修复/升级注意),预填到发布页面。月度成本约 0.5 元。

- 关键能力：commit `#Zentao编号` 自动关联,生成 3 段式 Markdown
- 响应时间：\<3 秒
- 降级：GitLab 不可达 / LLM 超时 / 格式异常 → 文本框为空,管理员手动输入

**AI-02 产品描述智能生成(P1)**
管理员审核新产品时,点击 `[AI生成描述]` 按钮,系统将组件名和 groupId 传给 DeepSeek,2-3 秒返回 30 字以内中文描述。按需调用,非自动触发。单次成本约 0.005 元。

#### 2.5.2.2 智能交互类(2 项)

**AI-04 自然语言查询(NL→SQL,P0)**
管理员在文本框输入自然语言(如"西安交付部门上个月下载了哪些产品"),LLM 翻译为 SQL,经 AST 净水器(sqlglot)校验后,以物理只读账号执行,表格展示结果。底栏显示查询耗时和返回行数。

- 关键能力：Schema Prompt 脱敏(仅暴露白名单表)+ AST 结构验证 + PG 只读沙箱
- 安全机制：每次查询写入 audit_log(原问题 + SQL + 返回行数)

**AI-05 自然语言搜索(下载门户,P1)**
搜索框输入"能导出 Excel 的报表"→LLM 匹配产品。DeepSeek 可用时语义搜索,不可用时 LIKE 关键词降级。

- 搜索策略：先尝试 SQL 全文搜索,仅 LIKE 无结果时才调用 LLM
- 响应时间：\<2 秒

#### 2.5.2.3 智能分析类(2 项)

**AI-03 健康度评分 AI 分析(P1)**
制品多维评分雷达图(下载活跃度/安全漏洞数/License 合规/版本时效),可选 AI 分析面板(LLM 解释评分依据+改进建议)。

**AI-06 统计看板智能洞察(P2,预留)**
未来可在图表下方增 LLM 生成的一句话洞察。当前章节仅保留接口,UI 暂不实现。

#### 2.5.2.4 AI 治理类(2 项)

**AI-07 LLM 熔断器管控(P0)**
显示当前熔断器状态(CLOSED/OPEN/HALF_OPEN)、失败计数、最近错误日志。super_admin 可手动重置。5 次失败自动 OPEN 300s,熔断期间全量降级。

**AI-08 Prompt 热加载管理(P0)**
提示词模板列表(按场景编码分组),在线编辑→保存后立即生效(无需重启),支持降级策略配置(fallback_enabled/fallback_behavior)。所有提示词存储在 `prompt_template` 表,禁止硬编码在 Python 源码中。

### 2.5.3 防护与治理体系

| 防护层 | 机制 | 目的 |
| ------------ | ------------------------------- | ----------------------------------- |
| 物理超时 | `asyncio.wait_for(timeout=10s)` | 防止 LLM 慢响应拖垮 API |
| 并发限流 | `asyncio.Semaphore(5)` | 第 6 个请求降级,保护 DeepSeek 配额 |
| 短路熔断 | 5 次失败 → OPEN 300s | 熔断期间全量降级,避免雪崩 |
| Schema 脱敏 | System prompt 仅暴露白名单表 | 从源头减少 Prompt 注入攻击面 |
| AST 净水器 | sqlglot 解析 + 黑名单表拦截 | 阻断非 SELECT 语句,提前拒绝恶意请求 |
| 物理只读沙箱 | PG `amdp_readonly` 账号 | 防止大模型生成的 SQL 越权写入 |
| 审计追踪 | audit_log 记录原问题+SQL+行数 | 完整可追溯,即使只读池拥塞也不丢审计 |

### 2.5.4 技术特点

- **统一模型**：DeepSeek chat API(国产,中文能力强,极低成本)
- **优雅降级**：LLM 调用是增强功能不是必要条件,任何异常静默跳过
- **异步非阻塞**：所有 LLM 调用通过 `asyncio.create_task` 异步执行
- **Prompt as Data**：所有提示词存储在 `prompt_template` 表,修改后调 `POST /api/admin/prompt/refresh` 热加载,无需重启或发版
- **分级版本管理**：`is_active` 字段控制生效版本,支持 A/B 测试

### 2.5.5 与传统方案对比

传统企业制品管理平台(Nexus、JFrog、Harbor 等)均未内置大模型能力。amdp 首次将 LLM 作为\*\*"管理员的文字助手"**嵌入发布管理和产品管理流程,填补了制品管理领域**"元数据质量靠人工维护"\*\*的长期痛点。

## 4.3 发布通知增强(附件+邮件)

发布新产品版本时，系统自动将release_notes渲染为PDF附件，随通知一并提供给交付端。邮件通知发送给被授权交付中心的所有成员。

### 4.3.1 通知附件生成

- 管理员创建发布后自动执行

- 1.从release_bundle.release_notes获取Markdown格式更新说明(各中心自由撰写)

- 2.通过weasyprint将Markdown渲染为PDF，命名为{product_code}-{version}-release-notes.pdf

- 3.PDF存入本地/data/release-notes/目录，路径写入notification.attachment_path

- 4.站内通知展示附件下载按钮

技术选型：weasyprint——Python原生库，pip install即可，不依赖外部二进制文件。

### 4.3.2 邮件通知集成

接入公司SMTP邮件服务，发送通知给被授权交付中心的所有成员。邮件含摘要+完整链接。

更新摘要：

- 数据采集引擎升级到v2.3.1，性能提升30%
- 报表引擎支持自定义模板

产品下载：https：//amdp.tjhq.com/portal/products/data-platform

邮件发送逻辑：查auth获取已授权部门→查user获取活跃成员→BCC批量发送。异步操作，不阻塞发布流程。

| ------------------ | -------- | -------------- | ----------- |

### 4.3.5 发版说明PDF的生命周期

PDF随release_bundle存在——只要发布包未被物理删除，PDF就不删除。产品下架时PDF保留(历史追溯)。目录结构：/data/release-notes/{product_code}/{version}/release-notes.pdf。Docker部署注意：目录需挂载为volume。

### 4.3.6 release_notes自动预填机制

遵循"自动预填+人工可编辑"原则。管理员创建发布时：

- 系统检查该产品是否关联了GitLab仓库(product_catalog.gitlab_repo_id)
- 若有关联：调用GitLab API读取CHANGELOG.md，提取对应版本段落→自动预填release_notes
- 若无关联或读取失败：文本框空白，管理员手动填写
- 无论何种来源，管理员可自由编辑release_notes内容

### 4.3.7 手动上传额外附件

管理员创建发布时可选择上传一个额外附件(截图、测试报告、部署手册等)，若需多个文件，人工压缩打包为ZIP后上传。

release_bundle表新增字段：extra_attachment_path(VARCHAR 500，NULLABLE)——管理员手动上传的附件路径。与release_notes生成的PDF附件独立存储。交付端通知中展示两个下载链接：[下载发版说明PDF]和\[下载附件\](仅在有附件时显示)。

### 4.3.8 LLM辅助生成release_notes

制品管理集成DeepSeek大模型能力，在管理员创建发布时自动生成发版说明草稿，管理员审核编辑后确认发布。

#### 4.3.8.1 触发时机与流程

- 管理员选择产品和各组件版本，点击"创建发布"
- 检查product_catalog.gitlab_repo_id
- 若有：调GitLab API获取两次release tag间的commit messages
- 将commit信息发送给DeepSeek，请求生成结构化发版说明
- LLM返回Markdown草稿，预填到文本框
- 管理员审核编辑后确认发布
- 若无GitLab关联或LLM调用失败：文本框为空，不阻塞发布

#### 4.3.8.2 Prompt模板

你是一个软件发版说明撰写助手。根据以下Git commit信息，生成简洁的发版说明。
用Markdown格式，包含：##新增、##修复、##升级注意。中文撰写，每行不超过80字。
产品：{product_name}
版本：{release_version}
Git commit记录：
{commits}
请生成发版说明：

#### 4.3.8.3 模型选择与配置

LLM_API_BASE=https：//api.deepseek.com/v1
LLM_API_KEY=${LLM_API_KEY}
LLM_MODEL=deepseek-chat
LLM_ENABLED=true # 可随时关闭

#### 4.3.8.4 异常处理

LLM调用是增强功能不是必要条件。任何异常都不阻塞发布：GitLab API不可达→跳过;LLM超时(5秒)→放弃;LLM返回异常→丢弃;LLM_ENABLED=false→不调用。所有异常均记录INFO级别日志。

#### 4.3.8.5 Zentao上下文增强(可选)

ZENTAO_BASE_URL=https：//rdm.tjhq.com
ZENTAO_TOKEN=${ZENTAO_TOKEN}

这个增强的价值：commit message的#XXX是公司已有的工作习惯，无需改变研发流程，只通过自动提取和API查询放大其价值——从"人看得懂的引用"变成"LLM写得出的上下文"。

### 4.3.9 LLM辅助生成产品描述(P1)

管理员审核新产品时，点击[AI生成描述]按钮，系统将该产品的组件名和groupId传给DeepSeek，2-3秒返回30字以内中文描述。按需调用，非自动触发。

交互：前端loading动画→返回预填→管理员点击"采纳"或"重新生成"。超时5秒。无缓存(成本低于缓存维护复杂度)。

## 4.4 产品与版本状态机

### 4.4.1 产品状态机（product_catalog.status）

```
                    ┌─────────┐
          Nexus发现  │ 待审定   │  管理员点击"忽略"
         ──────────►│(pending) │──────────────┐
                    └────┬─────┘              │
                         │                    ▼
                    管理员点击"发布"      ┌─────────┐
                         │              │ 已忽略   │
                         ▼              │(ignored) │
                    ┌─────────┐         └─────────┘
                    │ 已发布   │
                    │(released)│
                    └────┬─────┘
                         │
              管理员点击"下线" / CVE触发自动下线
                         │
                         ▼
                    ┌─────────┐
                    │ 已下线   │──── 管理员点击"重新发布" ────► 已发布
                    │(deleted) │
                    └─────────┘
```

**状态转换规则**：

| 当前状态 | 可转换至 | 触发者 | 条件 |
| -------- | -------- | ------------------- | -------------------------------- |
| 待审定 | 已发布 | center_admin | 所有组件在Nexus存在 |
| 待审定 | 已忽略 | center_admin | 标记非产品目录组件 |
| 已发布 | 已下线 | center_admin / 系统 | 手动/ CVE CRITICAL自动 |
| 已下线 | 已发布 | center_admin | 重新发布（创建新release_bundle） |
| 已忽略 | （终态） | — | 不再出现在管理列表中 |

**与 release_bundle 的关系**：产品状态≠版本状态。产品"已发布"表示至少有一个 release_bundle 状态为 GA。产品"已下线"表示所有 release_bundle 均已下线。一个产品可以同时有 GA 版本（最新发布）和 Deprecated 版本（旧版本但仍有授权部门在用）。

### 4.4.2 版本状态机（release_bundle.status）

```
                    ┌─────────┐
         管理员发布   │   GA    │  （正式发布版本）
         ──────────►│         │
                    └────┬─────┘
                         │
              管理员标记过时 / 新GA版本发布
                         │
                         ▼
                    ┌─────────┐
                    │  Beta   │── 管理员发布新版本 ──► GA
                    │         │
                    └────┬─────┘
                         │
              管理员标记废弃（仍可下载）
                         │
                         ▼
                    ┌──────────┐
                    │Deprecated│
                    │          │
                    └────┬─────┘
                         │
              管理员下线 / CVE触发
                         │
                         ▼
                    ┌──────────┐
                    │QUARANTINED│ ← OSV.dev 扫描到 CRITICAL/HIGH → 自动封仓
                    │(已隔离)   │   （下载阻断，需 super_admin 白名单特批）
                    └────┬─────┘
                         │
              管理员确认下线 / 白名单放行
                         │
                         ▼
                    ┌─────────┐
                    │ 已下线   │  （下载门户完全隐藏,已下载用户收到紧急通知）
                    │(revoked) │
                    └─────────┘
```

**状态转换规则**：

| 当前状态 | 可转换至 | 触发者 | 下载行为 | 通知 |
| ----------- | ---------------- | ----------------------- | --------------------------------- | ---------------------- |
| —（新建） | GA | center_admin | 正常下载 | new_release 通知 |
| —（新建） | Beta | center_admin | 不授权默认可见 | 无 |
| GA | Deprecated | center_admin | 仍可下载（灰标） | version_deprecated |
| GA / Beta | QUARANTINED | 系统（OSV.dev 扫描） | [LOCKED] 禁止下载（红底白字） | security_lockdown 通知 |
| QUARANTINED | 已下线 | super_admin | 完全隐藏 | emergency 通知 |
| QUARANTINED | GA（白名单放行） | super_admin + audit_log | 正常下载（is_risk_accepted 标记） | 无 |
| Beta | GA | center_admin | 正常下载 | new_release 通知 |
| Deprecated | 已下线 | center_admin / 系统 | 完全隐藏 | emergency 通知 |
| Beta | 已下线 | center_admin | 完全隐藏 | emergency 通知 |

**Beta 与 GA 的区别**：Beta 版本不触发 new_release 通知，不在下载门户默认列表中展示（需手动搜索），用于内部验证。GA 是面向交付端的正式版本。

## 4.5 关键模块异常分支处理

以下针对六个核心模块，描述异常路径的完整处理逻辑，与 3.4 的正常路径时序互补。

### 4.5.1 产品同步异常

| 异常 | 触发条件 | 处理逻辑 | 恢复方式 |
| -------------- | ------------------ | --------------------------------------------- | ---------------- |
| Nexus不可达 | 网络中断/Nexus宕机 | 跳过本轮，日志WARNING，下次整点自动重试 | Nexus恢复后自动 |
| Nexus返回500 | Nexus内部错误 | 同上，连续3次失败→写入notification告警管理员 | Nexus修复后自动 |
| 单仓库查询超时 | 某中心仓库数据量大 | 跳过该仓库，继续同步其他仓库 | 下一轮重试该仓库 |
| 组件已被删除 | Nexus中已不存在 | DB中product_catalog.status='已删除'（软删除） | 管理员可手动恢复 |
| DB写入失败 | PG不可达/磁盘满 | 日志ERROR，终止本轮同步 | PG恢复后下轮自动 |
| 组件超过1000条 | 仓库增长超过预期 | 分页拉取（pageSize=200），循环至空页 | 自动 |

**降级状态表**：

| 连续失败次数 | 行为 |
| ------------ | ------------------------------------------------------- |
| 1-2次 | 静默跳过，仅日志记录 |
| 3-5次 | WARNING级别日志 + 写入notification（仅super_admin可见） |
| 6次以上 | ERROR级别日志，状态标记为INACTIVE，等待管理员介入 |

### 4.5.2 发布异常

| 异常 | 触发条件 | 处理逻辑 | 用户感知 |
| ----------------------- | ------------------- | ----------------------------------------------- | ------------------------- |
| 组件版本在Nexus中不存在 | 版本号写错/已被清理 | 返回422 VERSION_NOT_IN_NEXUS，列出缺失组件 | 错误提示，发布不创建 |
| 产品已存在同版本发布 | release_version重复 | 返回409 DUPLICATE_RELEASE | 提示"该版本已发布" |
| LLM API超时（5秒） | DeepSeek不可达 | 降级为空文本框，管理员手动填写 | 无感（release_notes为空） |
| LLM返回格式异常 | JSON解析失败 | 降级为空文本框 | 无感 |
| 邮件发送失败 | SMTP不可达 | 日志WARNING，通知仍创建（站内可见），邮件跳过 | 站内有通知，邮件无 |
| PDF生成失败 | weasyprint异常 | 日志WARNING，通知创建但无附件 | 通知有，附件无 |
| 通知批量INSERT失败 | PG异常 | 发布已创建，通知创建失败→回滚通知部分，发布保留 | 发布成功但无人收到通知 |
| DB事务失败 | PG不可达 | 整体回滚，发布不创建 | 500 INTERNAL_ERROR |

**事务边界**：`release_bundle INSERT` + `release_component INSERT` + `product_catalog.status UPDATE` 在同一事务中。通知和邮件在事务提交后异步执行，失败不影响发布。

### 4.5.3 下载异常

| 异常 | 触发条件 | 处理逻辑 | 用户感知 |
| ------------------------------ | ---------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| 未授权 | 用户dept_id不在auth表中 | 返回403 DOWNLOAD_NOT_AUTHORIZED | "您所在部门未被授权下载该产品" |
| 产品已下线 | product_catalog.status='已删除' | 返回404 NOT_FOUND | "该产品已下线" |
| 版本已下线 | release_bundle.status='已下线' | 返回404 NOT_FOUND | "该版本已因安全原因紧急下线" |
| Nexus某组件不可达 | 单组件网络异常 | 返回502 NEXUS_UNAVAILABLE，列出故障组件 | "部分组件暂不可用，请稍后重试" |
| 并发超载 (Fail-Fast) | 当前活跃下载数 ≥ MAX_CONCURRENT_DOWNLOADS | 直接拒绝请求，返回 429 RATE_LIMITED，附带 Header Retry-After： 10。不占用任何内存与数据库连接。 | "系统当前下载繁忙，请稍后重试"。若为 CLI 脚本下载，接收到 429 后将自动静默退避重试。 |
| 超大包截断保护 | 组件总和体积超出 DOWNLOAD_MAX_ZIP_SIZE (如5GB) | 为防止超大流式压缩导致 HTTP 长连接超时断开，拒绝执行 ZIP 流式打包。返回 413 PAYLOAD_TOO_LARGE，附带降级策略。 | "文件体积过大，已切换为组件列表逐个下载模式"。 |
| 并发下载限流 | 同一用户5秒内>10次 | 返回429 RATE_LIMITED + Retry-After：5 | "请求太频繁，请5秒后重试" |
| 客户端网络极差/中断 (背压阻断) | 交付现场 TCP 连接意外断开 / 接收速度无限趋近于零导致超时 | 捕获 ClientDisconnect 异常，立即掐断与 Nexus 建立的下游 HTTP 数据流，丢弃未发送的 Chunk，并强制释放内存与 Semaphore 信号量。 | 下载中断失败，需在网络恢复后重新发起请求。系统端无残留挂起连接。 |
| 生产部署 License 缺失 | `deploy_intent=production` 且 `requires_license=True` 但未提供 License | 返回 403 `LICENSE_REQUIRED`，提示"该版本生产部署需要 License" | 交付人员可改为选"否"（trial）仅下载 |
| 生产部署 License 无效 | `deploy_intent=production` 且 License 验签失败/过期/冲突 | 返回 403/409，含具体原因（expired/invalid/conflict） | 交付人员联系管理员更换 License，或改为 trial 下载 |
| 组件已失效（幽灵文件） | `release_component.nexus_status = 'deleted'` | 下载时跳过该组件，写 WARNING 日志。若全部组件均失效，返回 410 GONE | "部分组件已被 Nexus 清理，请联系管理员"。若全部失效则提示"版本不可用" |

**下载日志写入**：download_log 在流式响应开始后异步写入（`asyncio.create_task`），不阻塞下载流。写入失败不影响已开始的下载。

### 4.5.4 CVE导入异常

| 异常 | 触发条件 | 处理逻辑 | 用户感知 |
| -------------------- | -------------------- | -------------------------------------- | -------------------------- |
| JSON格式错误 | Trivy输出被截断 | 返回400 VALIDATION_ERROR，标注错误行 | CI日志中可见错误 |
| CVE已存在 | 同CVE同release已导入 | 跳过不重复通知（幂等），返回已存在数量 | 200 OK + skipped：N |
| release_bundle不存在 | 版本号不匹配 | CVE记录创建但关联为NULL（孤儿漏洞） | 管理员后台可见未关联漏洞 |
| 通知创建失败 | PG异常 | CVE记录已写入，通知创建失败→回滚通知 | 漏洞已记录但无人收到通知 |
| 邮件发送失败 | SMTP不可达 | 同发布异常 | 站内有红色紧急通知，邮件无 |

**幂等保证**：`vulnerability_alert` 表通过 `(cve_id, release_bundle_id)` 复合条件判重——`SELECT WHERE cve_id=$1 AND release_bundle_id=$2 AND notified_at IS NOT NULL`，已通知的不再重复。

### 4.5.5 授权管理异常

| 异常 | 触发条件 | 处理逻辑 | 用户感知 |
| ---------------- | ------------------- | ---------------------------- | -------------------------- |
| 重复授权 | 同一部门+产品已授权 | 返回409 AUTH_ALREADY_EXISTS | "该部门已有此产品授权" |
| 撤回已撤回的授权 | revoked_at已非NULL | 返回409 AUTH_ALREADY_REVOKED | "授权已被撤回" |
| 尝试重新激活 | 修改revoked_at=NULL | 拒绝，提示创建新auth记录 | "请重新授权而非恢复旧授权" |
| 授权产品不存在 | product_id无效 | 返回404 PRODUCT_NOT_FOUND | "产品不存在" |
| 授权部门不存在 | dept_id无效 | 返回404 DEPT_NOT_FOUND | "部门不存在" |

**授权不可恢复设计**：`auth` 表设计明确——`revoked_at IS NOT NULL` 即为终态。业务上需重新授权→创建新 `auth` 记录（新 `granted_by_user_id` + 新 `granted_at`），杜绝"先撤回后激活"的灰色操作。

### 4.5.6 产品划转异常

| 异常 | 触发条件 | 处理逻辑 | 用户感知 |
| ------------------------------ | --------------------- | ------------------------------------------ | ---------------------------- |
| 新旧中心相同 | owner_center_id未变 | 返回400 INVALID_TRANSFER | "产品已在目标中心" |
| 目标中心不存在 | dept不存在/非研发类型 | 返回404 DEPT_NOT_FOUND | "目标中心不存在或非研发中心" |
| Nexus Content Selector更新失败 | Nexus不可达 | 重试3次（间隔2秒），仍失败→UPDATE回滚+告警 | "划转失败，Nexus不可达" |
| 产品不存在 | product_code无效 | 返回404 PRODUCT_NOT_FOUND | "产品不存在" |

**事务边界**：`product_catalog.owner_center_id UPDATE` + `audit_log INSERT` 在同一事务。Content Selector 更新在事务外执行（Nexus API不支持事务），失败时回滚 DB 事务以保持一致性。

## 4.6 制品生命周期自动归档

无需人工介入的「僵尸版本」清理机制，解决 Nexus 历史版本无限堆积、无人知道能否删除的痛点。

### 4.6.1 内部逻辑

APScheduler 每周日凌晨 3：00 触发 `auto_archive_stale_releases()`。流程：

```
download_log（近180天）→ 反查 release_bundle WHERE status IN ('GA','Beta')
  └── 零下载 180 天 → status='Deprecated' + audit_log + notify center_admin

download_log（近210天）→ 反查 release_bundle WHERE status='Deprecated'
  └── 零下载 210 天 → status='已下线' + audit_log + notify 所有已下载部门
```

**关键约束**：

- 仅对 **自研制品** 生效（代理仓库的第三方依赖由 Nexus Cleanup Policy 另行管理）
- 若任何一个下载记录落在时间窗口内，**不会**触发归档
- 自动下线后下载门户完全隐藏，已下载用户收到紧急通知

### 4.6.2 状态转换规则

| 当前状态 | 条件 | 新状态 | 通知对象 |
| ---------- | ------------ | ---------- | -------------------------- |
| GA / Beta | 180 天零下载 | Deprecated | center_admin（管理员确认） |
| Deprecated | 210 天零下载 | 已下线 | 所有已下载部门（紧急通知） |

### 4.6.3 配置项

| 配置项 | 默认值 | 说明 |
| --------------------- | ------ | ---------------------------- |
| `AUTO_DEPRECATE_DAYS` | 180 | GA→Deprecated 阈值（天） |
| `AUTO_OFFLINE_DAYS` | 210 | Deprecated→已下线 阈值（天） |

### 4.6.4 异常分支

| 异常 | 处理 |
| ----------------------------- | -------------------------------------------------------------- |
| download_log 表为空（新系统） | 所有 release_bundle 视为「无下载」，仅警告日志，不触发归档 |
| 批量更新中途 PG 不可达 | 本次已执行的 UPDATE 不回滚（逐行独立事务），未执行的重试到下周 |
| 通知批量插入失败 | 状态仍已变更，通知失败写 WARNING 日志，不阻塞后续行 |

## 4.7 幽灵文件巡检（Nexus 物理存储对账）

### 4.7.1 问题起源

Nexus 原生 Cleanup Policy 可在物理层自动删除过期文件（如"30 天未拉取的 Snapshot""仓库存储上限"），也可能被管理员手动清理。AMDP 数据库中的 `release_component` 索引表不会感知这些底层变更——**交付人员在页面上看到组件，点击下载后 Nexus 返回 404，导致实施中断**。

§4.6「自动归档」仅基于下载日志判断活跃度，不检查 Nexus 物理文件是否存在。两者解决不同问题：自动归档清理 AMDP 自己该淘汰的记录，幽灵巡检修复 Nexus 物理变更导致的数据脱节。

### 4.7.2 巡检机制

APScheduler 每天凌晨 4：00 触发 `ghost_scan()`，最终一致性模式：

```
release_component（status=available，排除已禁用）→ 分批取 component_path
  └─ 对每个 path 发 HEAD 请求到 Nexus
       ├─ Nexus 返回 200 → 跳过（文件健在）
       └─ Nexus 返回 404 → UPDATE nexus_status = 'deleted'
           └─ 写入 audit_log（component_path + 发现时间）
```

**关键约束**：

- **HEAD 请求**，不下载实体文件，单组件开销 < 100ms
- 分批执行（每批 `GHOST_SCAN_BATCH_SIZE` 个），批间间隔 1s，避免对 Nexus 造成压力
- 仅扫描近 90 天内有下载记录的组件（`WHERE last_downloaded_at > NOW() - INTERVAL '90 days'`），减少无效请求
- 巡检完成后，统计本次发现删除数，若 > 阈值则发送 WARNING 通知给管理员

### 4.7.3 下载前拦截

下载服务在发起 Nexus 请求前，先检查 `release_component.nexus_status`：

```
GET /download → 查 release_component
  ├─ nexus_status = 'deleted' → 跳过该组件，打 WARNING 日志
  └─ 其他组件正常请求 Nexus 拉取
```

若本次下载全部组件均为 `deleted`，返回 410 GONE："该版本全部组件已被 Nexus 清理，请联系管理员"。

### 4.7.4 配置项

| 配置项 | 默认值 | 说明 |
| ---------------------------- | ------ | ------------------------------- |
| `GHOST_SCAN_ENABLED` | `True` | 幽灵巡检总开关 |
| `GHOST_SCAN_BATCH_SIZE` | `50` | 每批 HEAD 请求数量 |
| `GHOST_SCAN_RECENT_DAYS` | `90` | 仅扫描近 N 天有下载的组件 |
| `GHOST_SCAN_ALERT_THRESHOLD` | `10` | 单次发现删除数超此值→通知管理员 |

## 4.8 LLM 智能查询沙箱

基于 AST 语法树解析 + PostgreSQL 物理只读沙箱的双重刚性防御，解决自然语言转 SQL 场景下 Prompt 注入绕过正则防御、越权读取敏感表的风险。两层防线各司其职：

- **AST 层（快速门禁）**：用 sqlglot 解析 LLM 返回的 SQL 生成 AST，阻断非 SELECT 语句和黑名单表访问，提前拒绝恶意请求，不消耗 PG 连接。
- **物理沙箱层（终极防线）**：使用专用 `amdp_readonly` 数据库角色 + 独立连接池执行 SQL，PG 层已剥夺对 user/auth/audit_log 等敏感表的 SELECT 权限。即使 AST 层被绕过，PG 自身也会拒绝越权读取。

### 4.14.1 Schema Prompt 脱敏

LLM system prompt 仅暴露业务查询相关的**白名单表**，不泄露 user、auth、audit_log 等核心安全表的结构信息，从源头减少攻击面。

```text
# 脱敏后的 schema prompt
-- product_catalog： 产品目录
--   id, product_code, product_name, category, owner_center_id, status
-- release_bundle： 发布包
--   id, product_id, release_version, status, release_notes, released_at
-- release_component： 发布包-组件关联
--   id, release_bundle_id, component_id, commit_sha
-- product_component： 产品组件
--   id, product_id, component_type, group_id, artifact_id, version, file_size
-- download_log： 下载记录
--   id, release_bundle_id, dept_id, downloaded_at
-- dept： 部门表
--   id, name, type(研发/交付)
-- notification： 通知
--   id, dept_id, type, title, content, is_read, created_at
```

### 4.14.2 安全护栏矩阵

| 防线 | 阻断目标 | 实现机制 |
| ---------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AST 语法层阻断 | 非 SELECT 语句、访问黑名单表 (user/auth/audit_log)、UNION 注入 | sqlglot 解析 SQL→遍历 AST 节点→检测非 SELECT 根节点或黑名单表引用→应用层直接抛异常阻断 |
| 物理沙箱隔离 | 越权读取密码哈希、权限记录、审计日志 | 专用 `amdp_readonly` DB 角色，PG 层剥夺对 user/auth/audit_log/license_check_log/storage_snapshot/vulnerability_alert 六张表的 SELECT 权限。应用层通过独立 AsyncEngine (`DB_READONLY_URL`) 建立只读连接池 |
| 行数上限强制约束 | 全表扫描拖垮 DB | sqlglot 在 AST 末端覆写 LIMIT 节点为 50（处理子查询中已有 LIMIT 的绕过场景） |
| 超时与资源熔断 | 恶意笛卡尔积打满 CPU | 只读连接池会话初始化时强制 `SET statement_timeout = '5s'` |

### 4.14.3 只读连接池架构

```
nl_query_service
  │
  ├─ get_readonly_session()  → AsyncEngine(readonly_url)
  │    └─ amdp_readonly@PG  → 仅白名单表 SELECT
  │
  └─ get_session()  → AsyncEngine(rw_url)
       └─ amdp@PG  → 全权限（业务写操作）
```

`DB_READONLY_URL` 格式：`postgresql+asyncpg：//amdp_readonly：${DB_READONLY_PASSWORD}@${DB_HOST}：${DB_PORT}/${DB_NAME}`

### 4.14.4 审计

每次 LLM 查询通过 `get_session()` (RW) 写入 audit_log（原问题 + 生成的 SQL + 返回行数），与查询执行通道分离，确保即使只读连接池拥塞也不丢失审计记录。

## 5.2 管理后台关键页面交互

### 制品管理

**产品管理**：左侧分类树(按归属中心+业务类型分组)，右侧表格(产品名+状态颜色标签+组件数+最新版本)。待审定产品高亮黄色，管理员点击→查看详情→发布或忽略。划转操作：点击变更归属→下拉新中心→确认弹窗。

**发布管理**：创建发布包表单(选择产品+版本号+发布说明Markdown，LLM预填)。版本状态流转：GA→Beta→Deprecated，QUARANTINED 由防火墙自动触发不可手动设置。版本晋升： super_admin 将发布包推广到其他中心。点击发布包→查看发布详情。

**发布详情**：发布包概览卡片（产品名/版本号/发布时间/发布人/状态），组件清单表格（组件名/版本/commit_sha/Nexus路径），发版说明 Markdown 渲染，下载统计图表（按部门/时间维度），版本状态流转历史。

**版本晋升**：选择目标中心→确认推广→系统自动更新目标中心的授权和通知。晋升结果：成功/失败列表，失败时显示原因并支持重试。

**同步状态**：上次同步信息卡片（时间/结果/耗时/发现新产品数），同步历史列表（时间/结果/发现数/失败原因），手动触发同步按钮（确认弹窗），同步进度条（实时显示当前同步的仓库和组件）。

### 交付管理

**依赖防火墙**：两个视图——(1) 白名单签署页： 显示所有 QUARANTINED 版本列表，super_admin 选择版本→填写原因→签署放行（7天有效期，写入audit_log）。 (2) 阻断状态查询： 输入版本号→查看防火墙判定结果（CRITICAL/HIGH/MEDIUM/LOW/通过）。

**License 校验**：按产品/部门查询 License 激活历史，显示序列号、激活时间、绑定产品。

**版本保留策略**：规则列表(产品+保留版本数)，新增/编辑/删除规则。

**部署记录**：按版本/部门查询 CLI 部署台账，显示主机名、IP、部署时间、结果（成功/失败）、部署意图。

**漏洞告警**：漏洞列表（CVE编号/严重程度/关联版本/发现时间/处理状态），按严重程度筛选（CRITICAL/HIGH/MEDIUM/LOW），批量处理（标记已处理/忽略），点击查看漏洞详情（描述/影响版本/修复建议/CVE链接）。

### 仪表盘

**概览**：全局驾驶舱——三个核心卡片（今日下载量/活跃用户/告警数），外部系统健康状态灯（Nexus/GitLab/LLM/SMTP 绿/黄/红），关键业务事件时间线（最近发布/同步/告警）。

**下载统计**：三个概览卡片(本月总下载/活跃用户/最热产品)。柱状图按部门下载量，环形图按产品占比，折线图近12月趋势。可筛选时间/部门/产品。中心管理员只看自己中心数据。

**健康度评分**：制品多维评分雷达图(下载活跃度/安全漏洞数/License合规/版本时效)，可选 AI 分析面板（LLM 解释评分依据+改进建议）。

**存储容量**：Nexus 各仓库容量占比环形图、近30天增长趋势折线图、预计溢出告警。

**代理仓库**：扫描 Nexus 代理仓库中超过阈值未使用的依赖包，列表展示(包名/大小/最后访问时间)，支持一键标记清理。

**系统健康**：外部系统状态仪表盘——四个状态灯（Nexus/GitLab/LLM/SMTP），点击状态灯展开详情：当前延迟、最近错误率、最后成功时间、一键重试按钮。异常时红色闪烁，点击可查看详细错误日志。

### 组织管理

> **前端实现**：用户管理、部门管理、角色权限、审计日志、标签管理页面由 `@company/iam-kit` 共享组件库提供，AMDP 通过 `main.ts` 注入 IAM 客户端并在路由中挂载 `iamRoutes` 实现复用。AMDP 自身仅保留业务特有的授权管理（AuthMatrixPage）。

**部门管理**：左侧部门树（研发中心→交付部门层级），右侧表格（部门名+类型+管理员+用户数）。部门数据由 identity-module Master 本地管理。新增部门：填写部门名/类型/上级部门/管理员→提交，直接写入本地 identity_department 表。编辑部门：修改基本信息，支持拖拽调整层级→保存。删除部门：必须先转移下属用户和产品授权，否则弹窗提示依赖关系→确认后软删除。

**用户管理**：表格（用户名/角色/所属部门/状态/创建时间）。用户数据由 identity-module Master 本地管理。新增用户：点击新增→填写用户名/员工编号/姓名/邮箱/角色/所属部门→提交，直接写入本地 identity_user 表。编辑用户：修改角色/部门/状态（启用/禁用）→保存。删除用户：软删除（is_active=false）。重置密码：点击重置→跳转至Pocket-ID密码重置页面。角色继承：super_admin 可管理所有用户，center_admin 可管理本中心用户，dept_admin 可管理本部门用户。

**角色权限**：角色定义列表（角色名/描述/权限范围），点击角色查看权限配置面板——左侧模块列表，右侧勾选该角色可访问的模块/操作。新增角色：填写角色名/描述→勾选权限→保存。预设角色：super_admin（全部权限）、center_admin（本中心产品管理+用户管理）、dept_admin（本部门授权+用户管理）、delivery（仅下载门户）。

**授权管理**（AMDP 业务特有）：部门↔产品授权矩阵——左侧部门树，右侧产品列表，矩阵单元格显示授权状态（已授权/未授权/过期）。批量授权：选择多个部门+多个产品→设置生效时间→确认。授权详情：点击单元格查看授权历史、生效时间、授权人。撤销授权：点击撤销按钮→确认弹窗→记录审计日志。

### 审计与查询

**审计日志**（iam-kit）：管理操作历史一览，支持按操作者/操作类型/时间范围筛选，点击查看 before/after JSON 快照对比。

**自然语言查询**（AMDP 业务特有）：文本框输入自然语言(如"西安交付中心上个月下载了哪些产品")→ LLM翻译为SQL→预览生成的SQL→执行→表格展示结果。底栏显示查询耗时和返回行数。

**Trace 查询**（AMDP 业务特有）：显示当前请求的 Trace ID，支持输入 Trace ID 查看跨层调用链（Traefik→FastAPI→Nexus）。

### 系统设置

**LLM 熔断**：显示当前熔断器状态(CLOSED/OPEN/HALF_OPEN)、失败计数、最近错误日志。super_admin 可手动重置。

**Prompt 管理**：提示词模板列表(按场景编码分组)，在线编辑→保存后立即生效（无需重启），支持降级策略配置(fallback_enabled/fallback_behavior)。

**系统公告**：公告列表（标题/内容/发布时间/发布人/状态），新增公告（标题/内容/是否置顶/生效时间/失效时间），编辑/删除公告，置顶公告显示在所有页面顶部红色横幅。

## 5.3 下载门户关键页面交互

**产品目录**：搜索框模糊搜索→瀑布流展示已授权产品卡片。卡片内容：产品名+归属中心标签+最新版本+发布时间。点击→版本列表(状态标记：GA绿色/Beta蓝色/Deprecated灰色/QUARANTINED红底白字[LOCKED]/已下线红色)。下载按钮→弹出下载确认框，含两项选择：(1) **部署意图下拉框**——“是否需要 License？”选"是"视同生产环境部署，占用授权名额，生成实施台账 (`deployment_record`)；选"否"为普通包获取（测试/备份/研发），仅留存下载日志，不扣减授权。(2) **下载方式**——Web ZIP 或 CLI 命令一键复制。

**隔离阻断交互**：当版本状态为 QUARANTINED 时，下载按钮变为灰色锁定[LOCKED]，按钮下方显示红色横幅：“该版本因安全漏洞已被隔离，请联系管理员申请特批放行”。交付人员无法通过任何方式下载。管理员可在 /admin/firewall 页面签署白名单后，该版本恢复可下载状态（下载时 is_risk_accepted 标记为 True）。

**通知中心**：顶部导航栏通知图标+未读气泡数→点击展开下拉面板。紧急通知：红色横幅"版本vX已紧急下线，请立即升级"。

**下载历史**：个人下载记录列表（产品名/版本/下载时间/部署意图/下载方式），按产品/时间筛选，点击查看下载详情（文件大小/下载耗时/Nexus路径）。

**个人设置**：个人信息卡片（用户名/角色/所属部门/邮箱），头像上传（支持拖拽上传），密码修改（旧密码/新密码/确认新密码），通知偏好设置（邮件通知开关/站内通知开关/紧急通知弹窗开关），保存后立即生效。

# 第八章 安全设计

## 8.1 多层防护体系

| **层级** | **机制** | **防护目标** |
| --------------- | ----------------------------------------------------------------- | ----------------------------------------------------------------- |
| 网络层 | Traefik SSL终结+IP白名单(可选) | 防未加密传输 |
| 应用层-认证 | JWT(HS256)，过期8小时;bcrypt(cost=12) | 防未认证访问和密码泄露 |
| 应用层-授权 | role-based+product-based双重校验 | 防越权操作和未授权下载 |
| 应用层-输入 | Pydantic请求体验证+SQLAlchemy参数化查询 | 防注入攻击 |
| 应用层-智能查询 | AST 结构验证 (sqlglot) + 物理级只读数据库账号凭证 (amdp_readonly) | 防止大模型生成的恶意 SQL 绕过前端逻辑，读取密码哈希或造成删库宕机 |
| 应用层-限流 | 每IP 60req/min， 登录接口5req/min， 下载接口20req/min | 防暴力破解和资源滥用 |
| 基础设施层 | Nexus Content Selectors路径级权限 | 即使绕过应用层，Nexus层仍然拦截 |
| 数据层 | audit_log记录所有管理操作JSON快照 | 事后审计追溯 |

## 8.2 认证与授权流程

### 8.2.1 OIDC 认证流程（AUTH_METHOD=oidc）

```
用户请求
│
▼
Traefik 路由分发
│
▼
ForwardAuth(Pocket-ID)──未认证──→ 重定向到 pocketid.tjhq.com
│已认证
▼
Pocket-ID 返回 OIDC ID Token（含 sub）
│
▼
amdp 验证 ID Token → 查询 identity_mapping 表获取 global_user_id
│  （只读查询，不创建用户；用户由 identity-module 管理）
│
▼
查询 identity_user 获取用户信息 → 查询 amdp_user_extension 获取 AMDP 特有字段
│
▼
生成 amdp JWT（含 global_user_id + role + dept_code）
│
▼
携带 JWT?──否──→ 401
│是
▼
JWT有效?──否──→ 401(TOKEN_EXPIRED)
│是
▼
角色可访问?──否──→ 403(AUTH_NO_PERMISSION)
│  权限校验：check_permission(global_user_id, permission_code)
│  支持用户直接权限 + 部门继承权限
│是
▼
[下载请求专属]
部门已授权?──否──→ 403(DOWNLOAD_NOT_AUTHORIZED)
│是
▼
执行业务逻辑→写 identity_audit_event（审计事件）
```

### 8.2.2 本地认证流程（AUTH_METHOD=local，过渡期使用）

```
用户请求
│
▼
POST /api/auth/login {username, password}
│
▼
查询 identity_user 获取 global_user_id → 查询 amdp_user_extension.password_hash
│
▼
bcrypt 验证密码 → 失败 → 401(AUTH_INVALID)
│成功
▼
生成 amdp JWT（含 global_user_id + role + dept_code）
│
▼
后续流程与 OIDC 一致（权限校验 → 审计）
```

## 8.3 密码与令牌策略

| **策略项** | **配置** | **说明** |
| ------------ | -------------------------------------- | ------------------------ |
| 密码哈希 | bcrypt(cost=12) | 不可逆 |
| 密码最小长度 | 8字符 | 前后端双重校验 |
| JWT过期时间 | 8小时(480min) | 到期需重新登录 |
| JWT签名算法 | HS256 | 密钥环境变量注入 |
| SECRET_KEY | ≥32字符随机字符串 | openssl rand -hex 32生成 |
| API限流 | 60req/min/IP（通用），下载20req/min/IP | 登录5req/min/IP |
| CLI Token | 独立30天过期 | 仅下载API，禁管理操作 |

# 第九章 异常处理与降级策略

## 9.1 故障场景矩阵

| **故障场景** | **影响** | **系统行为** | **恢复方式** |
| -------------------------------- | ---------------------------------------- | ------------------------------------------------------------- | --------------------------------------------------- |
| Traefik不可达 | 所有服务不可达 | 临时端口暴露，各服务独立访问 | Traefik恢复后恢复统一入口 |
| Pocket-ID不可达 | 认证失败 | 过渡期回退本地认证；正式期显示认证服务维护中 | Pocket-ID恢复后自动恢复 |
| identity-module 内部错误 | 用户管理操作失败 | 记录 error_log，重试操作；本地查询仍可用 | 重启应用后恢复 |
| identity-module Provisioning失败 | Pocket-ID 用户数据延迟 | 记录 error_log，指数退避重试；本地管理正常 | 恢复后自动重新推送 |
| Nexus不可达 | 产品采集暂停，下载失败 | 下载返回502+Nexus维护中;同步跳过+记日志;管理后台黄灯警告 | Nexus恢复后自动恢复 |
| PG不可达 | 所有操作失败 | 返回503;请求被中间件拦截 | PG恢复后自动恢复 |
| Nexus响应超时(>120s) | 下载中断 | httpx 120s超时→504，不重试 | 用户重新发起下载 |
| CRON失败 | 新产品不自动出现 | 记错误日志+失败计数;连续3次→通知管理员 | 下次周期自动重试 |
| 下载并发超载 (Fail-Fast) | 活跃下载达 MAX_CONCURRENT_DOWNLOADS 上限 | 返回 429 + Retry-After： 10；CLI 脚本自动静默退避重试 | 等待其他下载完成后自动恢复 |
| 超大包截断保护 | 组件总和 > DOWNLOAD_MAX_ZIP_SIZE (5GB) | 返回 413 + 降级为组件列表逐个下载 | 用户改逐个下载 |
| 客户端网络中断 (背压阻断) | TCP 意外断开/接收极慢 | 捕获 ClientDisconnect，掐断 Nexus 下游流，释放 Semaphore | 用户网络恢复后重试 |
| 生产部署 License Gate 拒绝 | License 缺/过期/冲突 | 返回 403/409 + 具体原因；下载仍可改为 trial 模式降级 | 用户改选"否"仅下载 |
| Nexus 组件已被清理（幽灵文件） | release_component.nexus_status='deleted' | 下载时自动跳过已失效组件；全部失效→返回 410；每日巡检持续标记 | 管理员检查 Nexus Cleanup Policy，必要时重新推送组件 |
| 单组件下载失败 | ZIP部分缺失 | 跳过失败组件，继续打包;README.txt说明缺失组件 | 管理员检查Nexus;用户逐个下载 |
| Content Selector失误 | 合法用户被拦截 | 不影响(使用服务账号访问Nexus，对用户透明) | 管理员修复配置 |

## 9.2 降级与恢复

| **降级场景** | **触发条件** | **降级行为** |
| --------------------- | ------------------------------ | ---------------------------------------------- |
| ZIP流式打包→组件列表 | 包体积 > DOWNLOAD_MAX_ZIP_SIZE | 返回 413 + JSON 清单 + 各自下载链接 |

**计**：1代理+21托管+2组=24个仓库。

### 10.1.2 命名规则

- **中心后缀**：研发中心A→`a`，研发中心B→`b`，……研发中心G→`g`（小写单字母，与dept表type='研发'对应）

- **托管仓库**：`{类型}-center-{中心字母}`，如 `maven-center-a`
- **代理仓库**：`{类型}-proxy`，如 `maven-proxy`
- **组仓库**：`{类型}-group`，如 `maven-group`
- **严禁**：用中文、数字编号、部门名称命名仓库

### 10.1.3 Blob Store配置

| Blob Store | 关联仓库 | 软配额建议 |
| --------------------------- | ---------------- | ---------- |
| `blob-maven-proxy` | maven-proxy | 50GB |
| `blob-maven-center-{a..g}` | 各中心Maven托管 | 20GB/中心 |
| `blob-pypi-center-{a..g}` | 各中心PyPI托管 | 10GB/中心 |

每个hosted仓库独立Blob Store——方便按中心监控容量、独立清理。

## 10.2 CI/CD制品推送接入说明

### 10.2.1 对系统设计的影响

零影响。不关心制品是谁推、什么方式推——研发手动push、GitLab CI自动推、Jenkins自动推都一样。

### 10.2.2 CI推送操作指引

每个研发中心有专属Maven仓库，CI脚本中推送地址必须指向正确仓库(如http：//nexus：8081/repository/maven-center-a/)。中心A→maven-center-a，中心B→maven-center-b，以此类推。

GitLab CI示例：在.gitlab-ci.yml中配置deploy阶段，mvn deploy -DaltDeploymentRepository指向对应仓库URL，仅tag推送。Jenkins：withCredentials注入Nexus凭证，mvn deploy同逻辑。

### 10.2.3 版本命名约定

Git Tag：v主版本.次版本.修订号(如v2.3.1)。Maven version：去除v前缀。amdp release_version：保持Git Tag原样。SNAPSHOT：制品管理自动过滤。Beta/RC：制品管理自动过滤。

### 10.2.4 同步频率与实时性

CI低频推送(每周几次)→每小时CRON同步→数小时可见。CI高频推送→同上，需确保SNAPSHOT被过滤。要求实时→改CRON为Webhook。绝大多数场景CRON已足够。

### 10.2.5 CI推送故障排查

CI推送成功但看不到：等下一个整点/确认版本名未被过滤。CI推送401：检查Nexus凭证和Content Selector写权限。推错仓库：修正CI配置+amdp管理员改owner_center_id。

## 10.3 对上游系统的最低要求

制品管理不对研发端流程增加负担——零改造要求。以下为"不做也能用，做了更丝滑"的约定清单，分三级。

| # | 级别 | 要求 | 原因 |
| --- | ------------ | --------------------------------------------------- | ------------------------------------------ |
| 1 | **必须** | 推到正确的中心仓库（研发中心A→maven-center-a） | 推错→归错中心 |
| 2 | **必须** | 正式发布版本不带SNAPSHOT | 制品管理自动过滤，SNAPSHOT永不进入产品目录 |
| 3 | **必须** | Maven groupId创建后不可修改 | 改了视为全新产品 |
| 4 | **必须** | GitLab API可访问 | release_notes LLM自动预填需要读仓库 |
| 5 | **必须** | GitLab CI Variable配置NEXUS_USERNAME/NEXUS_PASSWORD | 推送制品+Content Selector写权限 |
| 6 | **强烈建议** | 版本号用语义化(v2.3.1) | 非v2.3.1-20260618-hotfix格式→版本列表混乱 |
| 7 | **强烈建议** | 打tag才触发制品推送(only：tags) | 避免每次MR合并都推→CRON发现大量噪声 |
| 8 | **建议** | CI Job ID作为build_id传参 | 计算变更前置时间。不传不影响核心功能 |
| 9 | **建议** | commit末尾带#Zentao编号(已有规范) | 自动提取增强LLM上下文 |
| 10 | **建议** | 维护CHANGELOG.md | 不写也不阻塞，管理员手动填release_notes |
| 11 | **建议** | Tag格式统一为v主版本.次版本.修订号 | 发布版本号直接复用 |

> 底线：唯一硬性要求是"推到正确的Nexus仓库"，其余全部是"不做也能跑，做了更丝滑"。

# 第十一章 与DDAP系统集成

## 11.1 两个系统的关系定位

| **维度** | **DDAP(已运行)** | **amdp(新建)** | **交集** |
| -------- | ------------------------------------- | ---------------------------------- | ---------------------- |
| 核心功能 | 研发效能度量：看板、数据底座、dbt分析 | 制品分发：产品目录、下载管控、授权 | 共享Nexus数据源 |
| 用户 | 研发管理层+效能团队 | 交付中心+研发中心管理员 | 研发中心管理员两套都用 |
| 数据流 | Nexus←采集→dbt建模→看板 | Nexus←采集→产品目录→下载门户 | Nexus制品数据 |
| 数据库 | DDAP PG+dbt staging | amdp PG | 可共享PG实例，不共享表 |
| 部署 | Docker Compose(已有) | Docker Compose(新建) | 可共Traefik入口 |

## 11.2 衔接模式：独立运行，API互调

### 11.2.1 衔接架构

两套系统各独立：代码仓库、数据库、Docker编排均独立。通过REST API互调，任一宕机不影响另一个。

### 11.2.2 衔接注意事项

1. 共享Nexus配置：地址+账号+密码统一由配置中心管理，不各自维护
1. 数据一致性窗口：采集频率不同→同一时刻产品版本数可能不一致，但业务含义不同无需强一致
1. 认证体系统一：短期共享JWT Secret，长期迁移Authelia
1. API互调优雅降级：任一不可达不阻塞主流程

## 11.3 整合模式：统一代码库与部署

### 11.3.1 整合架构

amdp作为DDAP子模块，共享代码仓库、PG实例(不同Schema)、Traefik入口、Docker编排。

### 11.3.2 整合注意事项

1. 71条DDAP规范约束（关键8条已在12.8详列）
1. 共享用户体系风险：推荐两套系统各自维护user表，通过Authelia做身份映射
1. 数据库Schema隔离：ddap和amdp两个独立Schema，Alembic各自管理
1. 共享前端入口：不同子域名（ddap.hq.com/amdp.tjhq.com）
1. dbt集成：amdp下载数据纳入DDAP四层建模
1. Docker Compose融合：amdp独立容器，共享postgres和traefik
1. 代码仓库：Phase1独立仓库→Phase2 Monorepo直接合并（跳过Submodule）

## 11.4 两种模式对比与选择建议

| **维度** | **衔接模式** | **整合模式** |
| ------------ | ----------------------- | ---------------------- |
| 开发独立性 | 完全独立 | 需遵守DDAP规范 |
| 部署复杂度 | 两套docker-compose | 一套编排 |
| 运维负担 | 两套监控/备份 | 统一运维 |
| 数据一致性 | 有采集窗口差异 | 同一PG可直查 |
| DDAP规范约束 | 不受约束 | 必须遵守71条 |
| 代码耦合度 | 零耦合 | 共享基础设施 |
| 升级风险 | 互不影响 | 一方变更可能影响另一方 |
| 适合阶段 | Phase1独立开发→快速上线 | Phase2成熟后评估整合 |

建议：Phase1衔接模式，独立开发快速上线。Phase2稳定3-6个月后评估整合——看amdp数据对DDAP效能度量的价值和团队对DDAP规范的掌握程度。

## 11.5 两种模式共通的硬性要求

- Nexus配置统一：地址+账号+密码由一个配置源管理
- 认证体系统一：短期共享JWT Secret，长期Authelia
- 数据库Schema隔离：ddap vs portal
- API调用优雅降级：不可达不阻塞
- CI/CD分离：构建部署流水线独立

## 11.6 amdp→DDAP dbt模型打通基础

### 11.6.1 设计目标

amdp的数据库设计从第一天起考虑与DDAP dbt四层建模体系兼容。上线后DDAP的dbt管线可直接从amdp schema拉取数据，无需二次清洗。

### 11.6.2 命名规范对齐

已遵守：主键id(BIGINT)、外键_id后缀、时间戳_at后缀、布尔is_前缀、JSONB快照。

### 11.6.3 dbt四层模型映射

stg_portal\_\_\* (10个表，1：1映射amdp schema)
int_release\_\_lifecycle (release+DDAP GitLab CI→变更前置时间)
int_product\_\_delivery_adoption (auth+download→授权-下载转化率)
int_deployment\_\_success_rate (download_log.deploy_result→部署成功率)

dim_product (产品维度+生命周期+归属+组件数)
dim_dept (部门维度+类型+人数)
dim_release (发布维度+版本+状态+build追溯)

fct_download_event (每次下载)
fct_release_event (每次发布)
fct_auth_event (授权变更，SCD2追踪)

### 11.6.4 数据一致性

同一PG实例：dbt直接跨schema查询，零数据延迟。不同PG实例：通过foreign data wrapper或dbt external source。

### 11.6.5 打通后的BI能力

三个维度看板：发布效能(频率/前置时间/通过率/健康度)、交付效能(采纳速度/趋势/热度/覆盖率)、质量信号(紧急下线/部署成功率/活跃度/留存)。覆盖DevOps四大黄金指标中的三个，唯一缺失的MTTR需从客户现场监控系统获取。

## 11.7 统一认证：Pocket-ID OIDC集成

### 11.7.1 为什么需要

当前两套系统各自维护用户。统一认证后：一个账号处处可用、离职一处禁用、未来系统一次对接。

### 11.7.2 推荐Pocket-ID

Pocket-ID——企业级统一身份认证平台，与 identity-module 深度集成。支持OIDC/OAuth2/SAML/SSO，提供统一用户管理、Provisioning、审计日志能力。

### 11.7.3 架构

Pocket-ID：hq→Client：ddap+amdp→Role：super_admin/center_admin/dept_admin/delivery→User映射。两套系统都重定向到Pocket-ID登录，各自验证OIDC ID Token公钥。

### 11.7.4 迁移路径

- Phase1：注册Pocket-ID Client，配置OIDC回调地址
- Phase2：amdp先行，新增Pocket-ID登录入口并存旧登录。**首次双轨登录时查询本地 identity_mapping 表获取 global_user_id**。
- Phase3：DDAP跟进
- Phase4：关闭旧认证，删除password_hash字段

**关键约束**：

- 匹配键使用 `global_user_id`（通过 identity-module 本地 identity_mapping 表解析），由 Master 端本地生成并维护
- 映射仅在**首次 Pocket-ID 登录**时执行一次，不覆盖已有映射
- 冲突场景（同 username、不同 Pocket-ID sub）需人工介入，防止账号冒用
- `password_hash IS NULL` 的用户**禁止**旧 JWT 登录——仅限 Pocket-ID 通道
- DDAP offline时降级：记录source_username，标记pending_identity，后续补映射

## 11.9 HA 部署适配（Patroni + 双节点应用容器）

为支撑 PostgreSQL Patroni 主从集群和双节点无状态应用容器，完成五项核心改造。

- 下载服务：异步生成器流式返回（OOM 防御）
- 智能查询：sqlglot AST 语法树净水器
- 定时任务：带熔断机制的轻量级探针
- 数据模型：is_production_deployment 字段
- 脚手架：AST 访问器上下文感知

# 第十二章 外部系统集成设计

amdp 依赖 4 个外部系统：Nexus（制品源）、GitLab（代码+CI）、DeepSeek（LLM 能力）、SMTP（邮件通知）。本章定义集成方式、认证策略、故障模式与降级策略——是 Ch7 接口契约的**工程实现配套**。

## 12.1 外部系统总览

| 系统 | 角色 | 协议 | 认证 | 故障影响范围 | 降级策略 |
| ------------------------- | ------------- | --------------------- | --------------------- | -------------------------- | ------------------------------------------------------------ |
| Traefik | 统一网关 | HTTP/TCP | 无（基础设施） | 所有服务不可达 | 临时端口暴露 |
| Pocket-ID | 统一鉴权 | OIDC/OAuth2 | Session | 认证失败 | 回退本地认证（过渡期） |
| identity-module（Master） | 本地身份源 | 本地 CRUD | — | — | 本地 identity\_\* 表查询，无外部依赖 |
| Nexus | 制品存储 | REST (HTTP) | Basic Auth | 同步停滞、下载不可用 | 核心下载路径无降级；浏览/搜索仍可用（30%可用率，详见18.2.2） |
| GitLab | 代码托管 + CI | REST (HTTP) + Webhook | Personal Access Token | release_notes 无法自动生成 | 管理员手动填写发版说明 |
| DeepSeek | LLM 推理 | REST (HTTP) | API Key | 3 个 AI 功能全部不可用 | LIKE 搜索兜底；release_notes 空白框 |
| SMTP | 邮件发送 | SMTP (TCP 25/465/587) | 账号密码 / 匿名 | 邮件通知不可达 | 入库 notification 不丢，站内通知正常 |

## 12.2 Traefik 统一网关

### 12.2.1 架构定位

Traefik 作为统一入口网关，负责：

| 职责 | 说明 |
| -------- | ------------------------------------------ |
| SSL 终止 | 统一 TLS 证书管理，后端走 HTTP |
| 路由分发 | 按域名/路径分发到 amdp、ddap、Nexus 等服务 |
| 认证前置 | ForwardAuth 中间件对接 Authelia |
| 安全防护 | 安全头注入、限流、IP 白名单 |
| 服务发现 | 自动发现 Docker 容器服务 |

### 12.2.2 路由规划

| 域名 | 服务 | 认证 | 说明 |
| --------------------- | ---------- | --------- | --------------- |
| `hub.tjhq.com` | amdp | Pocket-ID | 制品下载门户 |
| `amdp-admin.tjhq.com` | amdp-admin | Pocket-ID | 管理后台 |
| `ddap.tjhq.com` | ddap | Pocket-ID | DevOps 效能平台 |
| `nexus.tjhq.com` | Nexus | Pocket-ID | 制品仓库 |
| `pocketid.tjhq.com` | Pocket-ID | 无 | 统一登录页面 |

### 12.2.3 中间件链

```yaml
# Traefik 中间件配置
middlewares:
  pocketid:
    forwardAuth:
      address: http://pocketid:9091/api/verify?rd=https://pocketid.tjhq.com
      trustForwardHeader: true
      authResponseHeaders:
        - Remote-User
        - Remote-Groups
        - Remote-Name
  security-headers:
    headers:
      customResponseHeaders:
        X-Frame-Options: DENY
        X-Content-Type-Options: nosniff
        Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'
        Strict-Transport-Security: max-age=31536000; includeSubDomains
  rate-limit:
    rateLimit:
      average: 100
      period: 1s
  compress:
    compress: {}
```

### 12.2.4 安全配置

| 配置项 | 值 | 说明 |
| --------------- | --------------------------- | ---------------------- |
| TLS 版本 | TLS 1.2+ | 禁用 SSLv3/TLS 1.0/1.1 |
| 证书 | Let's Encrypt（或企业证书） | Traefik ACME 自动签发 |
| HSTS | max-age=31536000 | 强制 HTTPS |
| CSP | default-src 'self' | 限制资源加载 |
| X-Frame-Options | DENY | 防点击劫持 |

## 12.4 identity-module 嵌入式身份管理（Master）

amdp 采用 [Identity Service 设计方案](Identity_Service_Design.md) 中的**方案B（嵌入式 identity-module）**，以 **Master** 角色运行——amdp 作为身份数据的主源，本地管理用户和部门，支持完整的增删改查，不依赖任何外部身份系统。

> 若需与 DDAP 等其他系统共享身份数据，DDAP 可作为 Replica 从 AMDP 同步。详见 [Identity Service 设计方案 §A.6.1](Identity_Service_Design.md#a61-dpap-%E4%BD%9C%E4%B8%BA-replica-%E9%83%A8%E7%BD%B2%E5%9C%BA%E6%99%AF)。

### 12.4.1 架构定位

```
┌──────────────────────────────────────────────────────────┐
│  amdp（Master：identity-module role="master"）           │
│  ┌────────────────────────────────────────────────────┐  │
│  │  identity_user / identity_department / identity_*  │  │
│  │  ↑ 本地 CRUD，写 API 全部开放                      │  │
│  │  ↑ 通过 Provisioning 推送至 Pocket-ID              │  │
│  └────────────────────────────────────────────────────┘  │
│                          │                               │
│  ┌───────────────────────▼────────────────────────────┐  │
│  │  DDAP（Replica，可选）                              │  │
│  │  AMDP 用户变更 → DDAP 定时拉取增量同步              │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 12.4.2 集成方式

```python
from identity_module import register_identity

register_identity(
    app=app,
    role="master",                        # 身份主源
    db_session_factory=get_session,       # 共享 amdp 业务 DB session
)
# → 自动完成：
#   ① 在 amdp DB 中创建 identity_* 表（Alembic 迁移）
#   ② 挂载完整 REST /identity/v1/* 路由（含写 API）
#   ③ 启动 Provisioning 同步至 Pocket-ID
```

### 12.4.3 用户管理能力

amdp 作为 Master，具备完整的用户和部门管理能力：

| 操作 | 端点 | 说明 |
| -------- | ------------------------------------ | ------------------------------- |
| 新增用户 | POST /identity/v1/users | 填写用户名/姓名/邮箱/部门/角色 |
| 编辑用户 | PATCH /identity/v1/users/{id} | 修改角色/部门/状态（启用/禁用） |

### 12.4.4 数据初始化

支持三种方式导入初始数据：

| 方式 | 适用场景 | 说明 |

### 12.4.6 同步到 Pocket-ID

身份变更通过 Provisioning 自动推送至 Pocket-ID：

| 方向 | 方式 | 说明 |
| ---------------- | --------------------- | ------------------------------------------------------ |
| AMDP → Pocket-ID | Provisioning 事件驱动 | 用户创建/修改/禁用时自动推送 |
| 推送失败 | 重试 + 标记 | 指数退避重试（5s/30s/120s），超限标记 permanent_failed |

### 12.4.7 健康检查

`/api/health` 端点独立探测 identity-module 状态，2秒超时：

| 状态 | 含义 | 影响 |
| ------------------- | ---------------------------------- | -------------------------------- |
| `identity=healthy` | 正常 | 无 |
| `identity=degraded` | Provisioning 推送至 Pocket-ID 失败 | 本地管理正常，Pocket-ID 同步暂停 |

### 12.5.1 连接配置

```
NEXUS_MAX_RETRIES=3       # 最大重试次数
NEXUS_RETRY_DELAY=2       # 初始重试间隔（秒）,指数退避：2/4/8
```

### 12.5.2 调用端点

| 接口 | 用途 | 调用方 | 频率 | 幂等 |
| ---------------------------------------------------------------------------------------------- | -------------------- | ------------- | ------------------- | -------------- |
| `GET /service/rest/v1/search?repository=maven-releases&maven.groupId=*&maven.artifactId=*` | 搜索全部组件 | 产品同步 CRON | 1次/小时 | 是 |
| `GET /service/rest/v1/search?repository=maven-releases&maven.groupId={g}&maven.artifactId={a}` | 搜索特定组件 | 发布时校验 | 按需 | 是 |
| `GET /repository/maven-releases/{path}` | 下载单个组件 | 下载服务 | 按需（每次下载N次） | 是 |
| `PUT /service/rest/v1/security/content-selectors/{name}` | 更新Content Selector | 产品划转 | 按需（每次划转1次） | 是（覆盖更新） |

### 12.5.3 故障模式

| 故障 | 检测方式 | 响应 | 恢复后行为 |
| ------------------------ | ---------------------- | ---------------------------------------------------------------------- | --------------------------- |
| 连接超时 | `httpx.ConnectTimeout` | 返回502 + "Nexus不可达"，客户端重试 | 下次请求自动恢复 |
| 读取超时 | `httpx.ReadTimeout` | 同上 | 同上 |
| 5xx 服务端错误 | HTTP status ≥ 500 | 退避重试3次，仍失败返回502 | 下次请求自动恢复 |
| 404 组件不存在 | HTTP 404 | 下载时跳过该组件（见4.4.3） | — |
| Basic Auth 过期 | HTTP 401 | ERROR日志，暂停同步，发告警 | 运维更新token后手动重启同步 |
| Content Selector更新失败 | HTTP 4xx/5xx | 重试3次（2s/4s/8s，与全局 NEXUS_RETRY_DELAY 一致），仍失败审计日志标注 | 管理员手动重试划转 |

### 12.5.4 健康检查

`/api/health` 端点独立探测 Nexus 连通性（`GET /service/rest/v1/status`），2秒超时。不可达时 `nexus=degraded`，product_sync 标记为不可用。

| 故障 | 处理 | 影响 |
| ------------------- | ---------------------------- | -------------------- |
| 连接超时 (>5s) | 立即放弃，fallback到降级策略 | 对应功能降级 |
| 返回429 Rate Limit | 等待3秒重试1次，仍失败则降级 | 短时不可用 |
| 返回5xx | 降级，WARNING日志 | 同上 |
| 返回乱码/格式化失败 | 降级，WARNING日志 | 同上 |
| API Key过期 (401) | ERROR日志 + 全部LLM功能暂停 | 用户感知AI功能不可用 |

| 策略 | 实现 |
| ------------ | ------------------------------------------------------------------- |
| 按需调用 | 不由CRON自动触发，仅用户手动操作时调用 |
| 结果缓存 | 同产品+同版本release_notes生成一次，缓存于release_bundle |
| token上限 | 输入token由调用方控制（commit messages ≤4000 chars），输出token=200 |
| 二级搜索策略 | 先尝试SQL全文搜索，仅LIKE无结果时才调用LLM |

### 12.8.1 连接配置

```

```

### 12.8.2 发送策略

| 策略 | 说明 |
| -------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 异步发送 | 使用FastAPI `BackgroundTasks`，不阻塞HTTP响应 |
| 批量发送 | 单次最多~700封（4中心×每中心~175人），分批发送（每批 200），整体 30 秒超时 |
| 失败重试 | 后台任务每10分钟扫描 `notification WHERE email_sent=FALSE AND created_at > NOW()-INTERVAL 24 HOURS`，批量重试（每批最多50封） |
| 发送确认 | 每封邮件发送后 `UPDATE email_sent=TRUE, email_sent_at=NOW()` |

### 12.8.3 故障模式

| 故障 | 检测 | 处理 |
| ---------------------- | --------------------------------- | -------------------------------------------- |
| SMTP连接不可达 | `smtplib.SMTPConnectError` | notification入库但email_sent=FALSE，后台重试 |
| 认证失败 (535) | `smtplib.SMTPAuthenticationError` | ERROR日志，暂停发送，发告警 |
| 单封发送超时 | 单封超时5秒 | 记录该收件人失败，继续下一封 |
| 整体超时 | 批量发送30秒超时 | 已发送标记成功，未发送标记失败等待重试 |
| 收件人邮箱不存在 (550) | `smtplib.SMTPRecipientsRefused` | 标记该条notification为"bounced"，不再重试 |

### 12.8.4 健康检查

`/api/health` 端点尝试SMTP连接（不发送邮件），2秒超时。不可达时 `smtp=degraded`。

# 第十四章 日志与监控设计

本章定义可观测性三支柱——**日志、指标、告警**——的系统级架构。14.6 已定义日志格式与级别约束，本章聚焦采集管道、监控指标、告警规则和看板设计。

## 14.1 可观测性架构

```
amdp 容器
   ├── stdout/stderr (结构化 JSON 日志)
   │       │
   │       ▼
   │   [Docker logging driver / Fluentd / Promtail]
   │       │
   │       ▼
   │   Loki / ELK（日志存储 + 检索）
   │
   ├── /metrics 端点（Prometheus 格式）
   │       │
   │       ▼
   │   Prometheus（指标采集 + 存储,15s scrape_interval）
   │       │
   │       ▼
   │   Grafana（看板 + 告警）
   │
   └── /api/health 端点（JSON,供 K8s liveness/readiness probe）
```

**技术选型**：Loki + Prometheus + Grafana（LPG 栈），理由：

- Loki 索引成本低（仅索引 label，不索引正文），适合日 GB 级日志
- Prometheus 生态成熟，FastAPI 通过 `prometheus-fastapi-instrumentator` 零侵入暴露指标
- Grafana 统一查看日志 + 指标，减少工具切换

### 14.2.4 Trace ID 链路追踪

所有请求必须携带 Trace ID，实现全链路追踪：

| 项目 | 说明 |
| -------- | ------------------------------------- |
| 生成位置 | 入口中间件 `middleware/trace.py` |
| 格式 | UUID4 |
| 传递方式 | HTTP Header `X-Trace-ID` |
| 传播范围 | API → Service → 外部调用（Nexus/LLM） |
| 日志字段 | `trace_id` 必须出现在所有结构化日志中 |

**Trace ID 传递示例**：

```python
# 中间件生成并注入
async def trace_middleware(request： Request, call_next)：
    trace_id = request.headers.get("X-Trace-ID", str(uuid.uuid4()))
    request.state.trace_id = trace_id
    response = await call_next(request)
    response.headers["X-Trace-ID"] = trace_id
    return response

# Service 层传递至外部调用
async def fetch_from_nexus(session, endpoint, trace_id)：
    async with httpx.AsyncClient() as client：
        return await client.get(
            f"{NEXUS_URL}{endpoint}",
            headers={"X-Trace-ID"： trace_id}
        )
```

```yaml
      - alert： AmdpDown

        for： 1m

        expr： amdp_db_connection_available == 0
        for： 1m

      # P1： 同步连续失败
      - alert： SyncDegraded
        expr： rate(product_sync_total{status="failed"}[15m]) > 0

          summary： "产品同步已连续失败 15 分钟"

      # P2： Nexus 响应异常
      - alert： NexusSlow
        expr： histogram_quantile(0.95, rate(nexus_request_duration_seconds_bucket[5m])) > 5
        for： 10m
        labels： {severity： P2}
        annotations：
          summary： "Nexus P95 延迟 > 5s"

      # P3： 磁盘预警
      - alert： DiskSpaceLow
```

## 14.7 全链路 Trace ID（分布式追踪）

2026-06-21 实现端到端 Trace ID 透传，跨越四层组件。

### 14.7.1 链路架构

```
浏览器(Vue/Axios)  →  Traefik/Nginx  →  FastAPI(contextvars)  →  Nexus(httpx)
  X-Request-Id=uuid     Access Log        async 安全透传            event hook 注入
     ↑                                                               ↓
  200 OK + header  ←  X-Response-Time  ←  [a1b2c3d4] INFO download_service
```

### 14.7.2 各层实现

| 层 | 文件 | 机制 |
| -------------- | ----------------------------- | -------------------------------------------------------------------------------- |
| **Vue 前端** | `frontend/src/utils/axios.ts` | `crypto.randomUUID()` 生成 + 拦截器自动注入；DEV 模式控制台打印耗时 |
| **FastAPI** | `app/middleware/trace.py` | `contextvars.ContextVar` 确保 async/await 安全；`TraceFormatter` 注入日志 |
| **main.py** | `app/main.py` | `app.add_middleware(TraceMiddleware)` + `TraceLogger.patch_root_logger()` |
| **Nexus 透传** | `app/infra/nexus_client.py` | httpx `event_hooks={"request"： [inject]}` 从 contextvars 提取 Trace ID |
| **Traefik** | `infra/traefik-trace.yml` | `customRequestHeaders： X-Request-Id： ""` 透传；`customResponseHeaders` 回写 |
| **Nginx** | 同上文件注释 | `proxy_set_header X-Request-Id $http_x_request_id` + `log_format combined_trace` |

### 14.7.3 排障示例

西安交付人员点击下载失败 → 浏览器 Console 显示 `[trace] a1b2c3d4 GET /api/download/TJHQ-XX/v2.3.1 502` → 运维 grep 网关日志 `grep a1b2c3d4 /var/log/nginx/access.log` → 发现 Nexus 超时 → 查到 Nexus 日志确认 `a1b2c3d4 HEAD /repository/maven-center-a/... 504` → 定位 Nexus 磁盘满。

## 14.8 轻量级 PLG 日志栈（Promtail + Loki + Grafana）

### 14.8.1 为什么不用 ELK

ELK（Elasticsearch + Logstash + Kibana）为全文索引而生，内存基线 2GB+。
AMDP"1 总部 + 4 分部"规模下，PLG 三件套内存总和 ~550MB，远低于 ELK。

| 对比 | ELK | PLG |
| ---- | ------------------- | --------------------- |
| 内存 | 2-4GB | ~550MB |
| 存储 | 全文索引（3x 膨胀） | 标签索引（原始大小） |
| 运维 | Java 堆调优 | Go 单二进制，开箱即用 |

## 14.9 大模型能力治理（Prompt as Data + 柔性降级）

### 14.9.1 隐患

系统 5 个功能点集成大模型（NL 查询、健康评分、发布公告、许可证解释、产品描述），
Prompt 硬编码在 Python 源码中 → 每次微调需重新发版。
DeepSeek API 宕机或限流 → 相关页面等待 60s 超时后卡死。

### 14.9.2 解决架构

```
产品/运营在后台修改 Prompt → prompt_template 表更新
    → 后台调 POST /api/admin/prompt/refresh → 清除 TTL 缓存
    → 下次 LLM 调用自动加载新 Prompt

LLM 调用流程：
  get_prompt(scene_code) → 查 DB (缓存 5min) → 填充 {var}
    → _query_deepseek()
      ├─ 成功 → 返回 LLM 结果
      └─ 失败 → fallback_fn() → 返回降级结果 + warning
```

### 14.9.3 组件清单

| 组件 | 文件 | 职责 |
| ----------- | ---------------------------------- | ------------------------------------------------------------------------------------------- |
| ORM 模型 | `app/models/prompt_template.py` | `scene_code` / `content` / `temperature` / `max_tokens` / `is_active` / `fallback_behavior` |
| Prompt 服务 | `app/services/prompt_service.py` | DB 加载 + TTL 缓存 300s + 内置默认兜底 |
| 降级包装器 | `app/services/llm_fallback.py` | `with_llm_fallback()` — 所有 LLM 调用唯一入口 |
| 种子数据 | `seed_prompt_template.sql` | 4 个场景的初始 Prompt（nl_query / health_score / release_notes / license_explain） |
| 消费者迁移 | `app/services/nl_query_service.py` | 硬编码 `_SCHEMA_PROMPT` → `with_llm_fallback(scene_code="nl_query", ...)` |

### 14.9.4 降级策略

每个场景的 `fallback_behavior` 决定 LLM 不可用时的行为：

| 行为 | 说明 | 示例 |
| ------------ | ---------------------- | ---------------------------------- |
| `rule_based` | 基于规则的默认计算 | 健康评分五维加权公式（不依赖 LLM） |
| `static` | 返回固定文本 | "发布公告暂不可用，请手动编写" |
| `cache` | 返回上次成功的缓存结果 | 同一产品代码的最近 3 次评估 |

### 14.9.5 运维操作

```bash

# 立即生效（无需重启！）
```

## 14.10 流量熔断与防雪崩（Circuit Breaker）

### 14.10.1 隐患

周一早高峰 DeepSeek API 60s 延迟 → AMDP 后端所有线程等待 LLM 响应被耗尽
→ 连"查询产品列表"都无法响应 —— 雪崩效应。

### 14.10.2 三条军规

| 机制 | 实现 | 参数 |
| ------------ | ------------------------------------- | -------------------------- |
| **物理超时** | `asyncio.wait_for(..., timeout=10.0)` | 超时即断，不盲等 |
| **并发限流** | `asyncio.Semaphore(5)` | 同一时刻最多 5 个 LLM 请求 |
| **短路熔断** | 5 次连续失败 → OPEN 状态 300s | 期间全部请求直接降级 |

### 14.10.3 状态机

```
        ┌───────────┐
   ───→ │  CLOSED   │ 正常通行，累计失败计数
        └─────┬─────┘
              │ fail>=5
              ▼
        ┌───────────┐
        │   OPEN    │ 拒接请求，直接降级 (5min)
        └─────┬─────┘
              │ 冷却到期
              ▼
        ┌───────────┐    试探成功     ┌───────────┐
        │ HALF_OPEN │ ──────────────→ │  CLOSED   │
        └─────┬─────┘                  └───────────┘
              │ 试探失败
              ▼
        ┌───────────┐
        │   OPEN    │ 重新计算 5min 冷却
        └───────────┘
```

### 14.10.4 代码结构

```
llm_fallback.py (唯一入口)
    │
    ▼
llm_circuit_breaker.call(fn=_query_deepseek, ...)
    │
    ├─ 状态检查（OPEN? → 直接 fallback）
    ├─ Semaphore.acquire() (并发限流)
    ├─ asyncio.wait_for(..., timeout=10) (硬超时)
    └─ 成功 → on_success / 失败 → on_failure → 状态迁移
```

### 14.10.5 配置项

| 配置 | 默认值 | 说明 |
| ---------------------- | ------ | --------------------------- |
| `HARD_TIMEOUT_SEC` | 10s | 单次 LLM 调用最大等待时间 |
| `MAX_CONCURRENT` | 5 | 同时进行中的 LLM 调用数上限 |
| `FAILURE_THRESHOLD` | 5 | 连续失败多少次触发熔断 |
| `RECOVERY_TIMEOUT_SEC` | 300s | 熔断后冷却时间 |

### 14.10.6 运维操作

```bash
# 查看当前熔断状态
# GET /api/admin/llm/breaker-status → {"state"："CLOSED","failures"：0}

# 手动重置
# POST /api/admin/llm/breaker-reset → {"state"："CLOSED"}
```

## 14.11 柔性降级实战（Graceful Degradation）

### 14.11.1 场景一：自然语言查询

| 状态 | 行为 | 用户感知 |
| -------- | ------------------------------------------------------------- | --------------------------------------------------------------------- |
| **正常** | LLM 翻译 NL→SQL → 返回结果 | 正常查询结果 |
| **降级** | `with_llm_fallback` 捕获异常 → 返回引导消息 | "🤖 智能助手当前正在开小差。已为您自动切换至高级过滤面板，请手动筛选" |
| **响应** | `{"degraded"： true, "hint"： "...", "results"： [引导消息]}` | 前端聊天框显示降级提示 + 手动筛选链接 |

### 14.11.2 场景二：制品健康度智能分析

| 状态 | 行为 | 用户感知 |
| -------- | ---------------------------------------------------------- | -------------------------------------------------------------------------- |
| **正常** | `/health-scores?with_ai_analysis=true` → LLM 生成诊断报告 | 五维面板 + 智能分析诊断长文本框 |
| **降级** | `generate_ai_analysis()` 返回 `None` | 仅展示五维数字面板 + 右上角黄点："当前为基础规则评分，AI 深度分析暂不可用" |
| **响应** | `{"scores"： [...], "analysis"： null, "degraded"： true}` | 前端 `v-if="analysis"` 隐藏分析面板 |

### 14.11.3 降级原则

- **不抛 500**：所有 LLM 失败路径吞噬异常，返回结构化退化结果
- **用户知情**：`degraded： true` + 明确提示当前状态
- **业务不中断**：规则引擎始终可用，LLM 仅作为增强层
- **前端配合**：`v-if="!degraded"` 控制智能分析面板显隐；`degraded=true` 时显示黄点标识

### 14.11.4 前端异步感知（局部骨架屏）

参见《AMDP 项目宪法 (contexts.md)》§2-§3。

- 绝对禁止全屏 Loading
- 基础数据瞬间渲染，AI 区块独立骨架屏
- 下载/查询/发布链路零 AI 依赖

Vue 范例：frontend/src/components/AiInsightPanel.vue

## 15.1 迁移工具

| 决策 | 选择 | 理由 |
| -------- | ----------------------------------------- | --------------------------------------------------------- |
| 迁移工具 | **Alembic** | SQLAlchemy 生态原生，自动生成迁移脚本，支持升级/降级/比对 |
| 迁移目录 | `migrations/`（Alembic） | 见 11.1.1 目录结构 |
| 迁移触发 | 应用启动时自动执行 `alembic upgrade head` | `main.py` 生命周期 startup 事件 |
| 执行顺序 | 按时间戳排序的 migration script 链 | Alembic 内置 `revision` 链 |

启动流程：

```python

from alembic import command

@app.on_event("startup")
```

## 15.2 迁移策略

| 场景 | 策略 | 工具 |
| -------- | ---------------------------------------------------- | --------------------------- |
| 首次部署 | `alembic upgrade head` 执行全部 migration chain | Alembic |

| 开发环境 | `alembic revision --autogenerate -m "描述"` 自动生成 | Alembic |
| 生产回滚 | `alembic downgrade -1` 退一个版本 | Alembic |

### 迁移脚本固化原则

- 所有 DDL 变更**必须**通过 Alembic 迁移脚本，**禁止**手动改数据库

- 每个迁移脚本标注 `upgrade()` 和 `downgrade()`，确保双向可逆
- CI 流水线中增加 `alembic check` 步骤，检测未提交的迁移

### 15.3.1 LLM 沙箱只读账号初始化

在首次部署 `alembic upgrade head` 执行完成后，系统自动执行以下基线权限固化（通过 Docker entrypoint 脚本或 Alembic data migration 实现）：

**账号创建**：

```sql
CREATE ROLE amdp_readonly WITH LOGIN PASSWORD '<DB_READONLY_PASSWORD>';
```

**白名单授权**：仅对制品查询强相关的业务表授予 SELECT 权限：

```sql
GRANT SELECT ON product_catalog, release_bundle, release_component,
                product_component, download_log, dept, notification
       TO amdp_readonly;
```

**黑名单防御**：坚决禁止以下核心机密表的读取权限授予该角色：

| 禁止表 | 原因 |
| --------------------- | ---------------------- |
| `user` | 防密码哈希泄漏 |
| `auth` | 防权限结构探索 |
| `audit_log` | 保护管理操作隐私 |
| `license_check_log` | 防许可证合规数据外泄 |
| `storage_snapshot` | 防存储架构探测 |
| `vulnerability_alert` | 防漏洞信息被非授权聚合 |

> **设计原则**：白名单 + 黑名单双保险。白名单仅授必要表，黑名单显式禁止敏感表。即使 DBA 误操作 GRANT ALL，黑名单表也在应用层 AST 校验中被拦截（4.14.2）。

## 15.5 版本升级迁移示例

假设 v1.1 版本新增 `release_notes_md` 列：

```bash
# 1. 开发环境生成迁移脚本
cd backend
alembic revision --autogenerate -m "add release_notes_md to release_bundles"

# 2. 检查生成的脚本
# alembic/versions/xxxx_add_release_notes_md.py
def upgrade()：
    op.add_column('release_bundles',
        sa.Column('release_notes_md', sa.Text(), nullable=True))
    # 将现有 release_notes 数据迁移到新列
    op.execute("UPDATE release_bundles SET release_notes_md = release_notes")

def downgrade()：
    op.drop_column('release_bundles', 'release_notes_md')

# 3. 测试升级
alembic upgrade head
# 验证列存在 + 数据完整

# 4. 测试回滚
alembic downgrade -1
# 验证列已删除 + 服务正常

# 5. 提交 Git → CI 自动执行
```

## 15.6 回滚策略

| 场景 | 回滚方式 | 注意 |
| ------------------ | ------------------------------ | ----------------------------------------------------- |
| 迁移脚本执行失败 | `alembic downgrade -1` | 在同一个事务中失败会自动回滚（PostgreSQL DDL 事务性） |
| 迁移成功但业务报错 | **不回滚数据库**——回滚代码版本 | 数据库向前兼容：新列允许 NULL |
| 灾难恢复 | PG 全量备份 + WAL 归档 → PITR | 每天凌晨全量备份 + 持续 WAL 归档 |

### 数据库兼容性约束

所有 DDL 变更必须满足**向前兼容**：

- 新增列：`DEFAULT` + `NOT NULL` 或允许 `NULL`
- 删除列：先标记 deprecated，大版本时统一清理
- 修改列类型：使用 `ALTER COLUMN ... USING` 显式转换
- 重命名列：**禁止**——通过新增列 + 数据迁移替代

## 15.7 迁移脚本检查清单

| # | 检查项 | 通过标准 |
| --- | ------------------------------------------- | ------------------------------------ |
| C01 | `alembic upgrade head` 成功 | 无异常退出 |
| C02 | `alembic downgrade -1` 成功 | 回退到上一版本 |
| C03 | 已有数据无损 | 迁移前后数据行数一致（除 seed 插入） |
| C04 | 索引/约束完整 | `\d table_name` 与预期一致 |
| C05 | `alembic check` 通过 | 无未提交迁移 |
| C06 | upgrade + downgrade + upgrade 连续 3 步成功 | 幂等性验证 |

## 17.4 结论：为什么自建

1. **场景错配**：Nexus Pro / JFrog 解决的是"研发团队内部的制品仓库管理"，华青真正的痛点是"研发→交付的制品传递断层"——市面上没有现成产品解决这个问题。
1. **用户错配**：现有产品的 UI/UX 面向开发者（Maven坐标、GAV、SNAPSHOT术语），华青交付人员不需要懂这些——他们只需要"搜→看→下"三件事。
1. **成本倒挂**：3人月开发投入 vs 60-100万三年License，ROI > 6：1。
1. **组织适配**：7研发中心+4交付中心的矩阵结构、产品跨中心划转——是华青独有的组织逻辑，任何商业产品无法原生支持。
1. **能力内建**：自建意味着后续可演进——CVE追踪、自然语言搜索、部署反馈闭环——这些差异化能力随组织成长，不受供应商Roadmap约束。

# 第十八章 竞品差异化与定位

第十七章回答了"为什么不自购"（理性比较），本章回答"amdp 到底是谁"（战略定位）。产品定位不是功能清单的堆砌，而是**选择做什么、选择不做什么、为谁而做**的决策集合。

## 18.1 定位陈述

> **amdp = Nexus 的对外窗口，不是 Nexus 的替代品。**
>
> 让需要制品的人——不必懂 Maven、不必问研发、不必翻邮件——就能找到、拿到、用上。

一句话定位：**企业级制品分发门户**。关键词拆解：

| 关键词 | 含义 | 对标缺失 |
| ------ | ------------------------------------------------- | ----------------------------- |
| 企业级 | 按组织架构授权，7中心+4交付中心矩阵映射 | Nexus Pro RBAC 不懂"中心"概念 |
| 制品 | Maven JAR/WAR/ZIP，不碰容器镜像（那是Harbor的事） | JFrog 试图做全类型→复杂且贵 |
| 分发 | 非研发→交付链路，非研发→研发链路 | 所有竞品都是研发→研发 |
| 门户 | 浏览器=客户端，无需安装任何工具 | 竞品必备 CLI/Maven 插件 |

## 18.2 差异化六支柱

| 支柱 | amdp | Nexus Pro | JFrog Artifactory | Harbor | 差异化程度 |
| ------------------------ | ----------------------- | ------------------ | ----------------- | ------------- | ----------------- |
| **分发优先 vs 存储优先** | 搜索→点击→下载 | 先知道Maven坐标 | 先建仓库 | `docker pull` | [RED] 根本性差异 |
| **非技术人员可用** | 浏览器三件事 | 登录后看原始XML | Web UI 仍面向Dev | 仅容器场景 | [RED] 根本性差异 |
| **组织架构原生映射** | dept_id 关联授权 | RBAC需手动映射 | RBAC需手动映射 | 项目级权限 | [YELLOW] 显著差异 |
| **中文AI辅助** | LLM搜索+发版说明 | [NO] | [NO] | [NO] | [YELLOW] 显著差异 |
| **通知闭环** | 发布→通知→下载→部署结果 | [NO] | 邮件(基础) | [NO] | [GREEN] 锦上添花 |
| **零学习成本** | 1分钟上手 | 需Maven/Gradle基础 | 需Artifactory概念 | 需Docker基础 | [RED] 根本性差异 |

## 18.3 多用户视角的价值主张

同一产品，不同角色看到不同的价值——这是"分发"区别于"存储"的核心。

| 角色 | 他们关心什么 | amdp 给什么 | 竞品给不了什么 |
| ----------------------- | ---------------------------------- | ----------------------------------- | ----------------------------------------------- |
| **交付人员**（~700人） | 怎么找到最新版本？怎么下载？ | 搜索→产品页→版本列表→一键下载ZIP | 需要先学 Maven 坐标、看懂 GAV、配 settings.xml |
| **中心管理员**（7人） | 我的产品谁在用？版本怎么管理？ | 发布→通知→统计→划转，全在同一个界面 | 发布需建 staging 库、通知靠邮件手动、划转不存在 |
| **部门管理员**（4人） | 我的团队需要哪些产品？权限怎么管？ | 授权面板→勾选产品→即时生效 | RBAC 配置脱离业务场景 |
| **超级管理员**（1-2人） | 全局大盘、成本、合规 | 审计日志+下载统计+运维看板 | 日志分散在多个系统，无统一视图 |

## 18.4 竞争护城河

为什么竞品不能轻易复制 amdp 的定位？四个不对称优势：

| 护城河 | 说明 | 竞品复制难度 |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **场景深度** | 分发场景需要理解华青"研发→交付"的具体工作流——交付人员怎么接到客户需求→怎么知道该部署哪个版本→怎么反馈部署结果。这不是通用功能，是领域知识。 | [RED] 极高——竞品没有华青的组织上下文 |
| **零边际成本** | 复用现有 Nexus OSS（已免费运行多年），amdp 只做管理层。竞品需要从底层仓库到上层门户全栈自研或并购。 | [YELLOW] 中——竞品可以加门户层，但商业模式不支持"免费加功能" |
| **组织变革资产** | 通知模板、角色定义、划转流程——都内嵌了华青 25 年的组织运作逻辑。竞品只能做通用 RBAC，无法做"研发一中心的产品划到研发三中心"这种组织特化操作。 | [RED] 极高——任何竞品都不可能为华青定制组织逻辑 |
| **中文AI能力** | DeepSeek 集成（中文commit理解、中文自然语言搜索）是华青内部场景驱动。Sonatype/JFrog 的 AI 能力（Nexus IQ / JFrog Xray）集中在漏洞扫描，非分发场景。 | [YELLOW] 中——技术上可能，但市场优先级不在中国中型企业场景 |

## 18.5 不做清单

有选择即有放弃。明确"不做"比"做什么"更能定义产品边界：

| 不做 | 为什么 | 谁在做 |
| ------------------- | ----------------------------------- | -------------------- |
| 不做制品存储 | Nexus OSS 已做好，没必要重新发明 | Nexus OSS |
| 不做容器镜像管理 | 规模小，`docker save/load` 已够用 | Harbor（如未来需要） |
| 不做 CI/CD 流水线 | GitLab CI 已运行多年，DDAP 已有集成 | GitLab CI + DDAP |
| 不做代码扫描/SAST | SonarQube 已在 DDAP 体系中 | SonarQube |
| 不做商业化/SaaS | 内部工具，不对外销售 | — |
| 不做多语言包/国际化 | 全中文组织，无国际化需求 | — |

## 18.6 定位一句话

> **"Nexus 管存储，amdp 管分发——一个管仓库，一个管快递。"**

在组织内推广时，这句话比任何功能列表都管用——交付人员一听就懂：Nexus 是研发的仓库（我不需要进去），amdp 是给我的快递包裹（我只需要签收）。

# 第二十章 组织变革与推广策略

产品做出来只是成功的一半，让 7 个研发中心 + 4 个交付中心**真正用起来**才是另一半。本章定义上线后的组织变革路径——谁先上、怎么推、阻力在哪、如何度量。

## 19.1 变革地图：从现状到目标

| 维度 | 现状（As-Is） | 目标（To-Be） | 变革幅度 |
| -------- | ---------------------------- | ------------------------------- | ---------------------------------------- |
| 产品发现 | 问研发→翻邮件→找群聊 | 搜索框→产品页→版本列表 | [RED] 大 |
| 制品获取 | 研发发网盘/邮件附件/QQ传文件 | 浏览器点击下载 ZIP | [RED] 大 |

## 19.2 干系人分析
