# 协作契约与高价值 Prompt 沉淀手册 (collaboration_patterns.md)

> **定位说明**：本文件是基于本项目长达 6 个多月、上千次人机互动的物理语料与行为特征提炼而成的**协作默契模型 (Collaboration Covenant)**。
> **核心目的**：沉淀用户的交互规律与高价值 Prompt 模板，帮助后续接手的 AI 助手瞬间达成“无缝心流对齐”，避免低效的重新磨合。
> **更新机制**：在每次会话交接或产生新的协作教训时，由 AI 自动追加沉淀到本文档。

## 一、 用户的语言特征与交互规律 (Interaction Dialect)

在长期的协作中，用户的输入呈现出极高密度的**工程化、确定性与敏捷性**特征。

### 1. 超极简驱动语流 (Ultra-compact Directives)

- **语料特征**：「继续」、「同意」、「OK」、「开始」、「干活」、「A / B / C」
- **行为语义**：用户倾向于通过最小的认知和输入摩擦，将系统直接推入物理执行态。
- **AI 继承要求**：**严禁在收到此类极简输入时进行长篇大论的废话回复**。AI 必须瞬间解码出隐藏的命令状态机，直接切入对应的物理动作（如修改代码、执行 pre-commit 或写入 progress.txt）。

### 2. 具象化风险审问 (Concrete Starvation Queries)

- **语料特征**：「...会不会炸啊？」、「...有没有埋坑？」、「会引发性能风暴/死锁风暴吗？」
- **行为语义**：用户通常在架构设计、DB模型变更、异步审计或大規模数据采集时抛出此类提问。
- **AI 继承要求**：这代表用户拉响了\*\*“物理拉垮预警”\*\*。AI 必须停止盲目自信，以极致的严谨态度，从“N+1 查询风暴”、“SQLite/PG 事务死锁”、“Pydantic 类型强校验”等物理底层出发进行根因排查，给出确凿的取证和防呆架构设计。

### 3. 概念契约化与“法条”升格 (Covenant Evolution)

- **语料特征**：「两阶段对齐协议」、「物理隔离」、「零日期崩溃」、「物理验证」
- **行为语义**：用户绝不容忍同类错误犯第三次。一旦排查出深层障碍，用户会要求将其抽象为某种“协议/红线”，并“落库/固化”到 `.agent/skills/` 的自动化脚本中。
- **AI 继承要求**：识别此类概念，必须遵循“文码同行律”，不仅修改代码，还要同步修改对应的 Markdown 规范并执行双向回标。

## 二、 复制即用的高价值 Prompt 模板 (High-Value Prompt Templates)

以下是互动中高频出现、具备极高审计与控制价值的 Prompt，用户（或后续 AI 自治）可直接复制使用：

### 1. 任务启动与卡点预判

```markdown
# 触发命令: /task-kickoff <任务描述>
或者直接发送:
"帮我评估下这个修改对系统架构的影响，有没有潜在的循环引用和 DB 锁风暴？"
```

- **价值所在**：强迫 AI 停止“编码冲动”，首先去拉取宿主机环境健康度、审查 `contexts.md` 技术锚点，做 L1-L4 的逻辑切片。

### 2. 演进技能与双向回标审计 (自动进化)

```markdown
"已经从 lessons-learned.log 提取并纳入 skill 的，是否在 lessons-learned.log 中已经标注，防止后续再重复？"
```

- **价值所在**：这是自进化循环的“最终门禁”，强迫 AI 去扫描已固化的 Skill 与原始 log，并在 `lessons-learned.log` 对应条目中强行追加 `[Codified into Skill: <skill-name> §<N> at <date>]` 的物理标记，消灭静态文档的腐烂。

### 3. 架构方案博弈 (A/B 案对照)

```markdown
"给出 A/B 两个技术路线，对比它们的性能瓶颈（尤其是并发和 DB 连接开销）、维护成本以及对现有 Staging 物理表的影响，推荐最稳妥的那一个。"
```

- **价值所在**：触发“询问锁（Inquiry Lock）”，要求 AI 进行结构化选项呈报，同时强迫其提供“AI 视角推荐路径”，减少用户的对齐成本。

### 4. 离场审计与清场

```markdown
"下班例行会话交接审计" 或 "准备下班/结束会话，执行 session-handover"
```

- **价值所在**：触发离场 DoD。强制执行 `just clean` 移除无害垃圾文件，在 `session-history.log` 置顶归档本次会话的 Key Deliverables 与 Next Steps，并在 `progress.txt` 中同步状态。

用户的上述习惯已被固化至 \[AGENTS.md\](file:///home/netxs/devops/AGENTS.md) 的多个核心红线中：

1. **意图映射 (AGENTS.md §1.1)**：
   - `"下班" / "吃饭" / "休息" ➜ /session-handover`
   - `"开始" / "上班" / "干活" ➜ /session-handover (Bootstrap)`
1. **执行锁与询问锁 (AGENTS.md §5)**：
   - 用户使用“？”结尾的疑问句时，**物理锁闭**所有写操作权限，AI 必须呈报方案并给出 A/B/C 结构化选项，等待用户选择。
   - 用户确认“开始/OK”后，AI 必须通过 `just progress-focus` 锁定重点，生成 `task.md` 拆解任务后，方可启动物理写操作。
